# Claude Codeのカスタムコマンドとは？スラッシュコマンドの作り方入門

「毎回同じプロンプトを入力するのが面倒...」「チームで指示の書き方がバラバラになっている...」と感じたことはありませんか？

今回伝えたいことの要点は、**カスタムコマンドを使えば、繰り返しのプロンプト入力から解放される**ということです。一言で言えば、「Markdownファイルを1つ作るだけで、`/コマンド名` で呼び出せる」というシンプルな仕組みです。

この記事では、カスタムコマンドの基本概念から、配置場所の使い分け、実際の作成手順までを解説します。読み終わる頃には、自分専用のコマンドを1つ作れるようになっているはずです。

## 忙しい人のために要約

この記事のポイントは以下のとおりです。

- **カスタムコマンド**は頻繁に使うプロンプトを `/コマンド名` で呼び出せる機能
- **プロジェクト用**は `.claude/commands/`、**個人用**は `~/.claude/commands/` に配置
- **ファイル名がそのままコマンド名**になる（`suggest-commit.md` → `/suggest-commit`）
- Markdownファイルにプロンプトを書くだけで作成完了
- コマンド追加後は `/clear` またはセッション再起動で反映

## カスタムコマンドとは

カスタムコマンドは、Claude Codeで繰り返し使う**プロンプトをコマンド化する機能**という役割を果たします。

たとえば「mainブランチとの差分を確認して、コミットメッセージを提案してください」というプロンプトを毎回入力する代わりに、`/suggest-commit` と入力するだけで同じ指示を実行できるようになります。そのため、入力の手間を大幅に削減できるという特徴があります。

さらに、プロンプトの内容を統一できるので、品質のばらつきを防げるでしょう。また、チームでコマンドを共有すれば、開発効率を向上させることも可能です。

ただ、カスタムコマンドは「**プロンプトのショートカット**」に過ぎない点には注意が必要です。複雑なロジックや条件分岐が必要な場合は、YAMLフロントマターやサブエージェントとの連携を検討してください。

YAMLフロントマターを使った高度なカスタマイズについては、別の記事で詳しく解説しています。

[Claude Codeカスタムコマンドのフロントマター設定：description・allowed-tools・model](https://zenn.dev/tmasuyama1114/books/claude_code_basic/viewer/custom-command-frontmatter)

!

**補足**：カスタムスラッシュコマンドはスキルと同じ扱いに統合されました。旧形式（`.claude/commands/`）は引き続きサポートされているため、既存のコマンドはそのまま使えます。詳しくは以下の記事をご覧ください。

[【Claude Code】カスタムスラッシュコマンドがスキルに統合されました](https://zenn.dev/tmasuyama1114/articles/cc_commands_merged_into_skills)

## カスタムコマンドの配置場所

カスタムコマンドには**2種類の配置場所**があります。用途に応じて使い分けることで、効率的に管理できるでしょう。

### プロジェクトコマンド

プロジェクトのルートディレクトリに `.claude/commands/` を作成し、コマンドファイルを配置してください。以下のような構成になります。

```
project/
├── .claude/
│   └── commands/
│       └── suggest-commit.md
└── src/
```

プロジェクトコマンドには**3つの特徴**があります。

- そのプロジェクトでのみ使用可能
- Git管理でチーム共有が可能
- プロジェクト固有のワークフローに最適

チーム開発で「全員同じ手順でレビューしたい」といった場面に向いています。

### 個人コマンド

ホームディレクトリに `~/.claude/commands/` を作成し、コマンドファイルを配置してください。

```
~/.claude/
└── commands/
    ├── daily-report.md
    └── my-review.md
```

個人コマンドの特徴は**汎用性**です。  

すべてのプロジェクトで使用できるため、「どのプロジェクトでも使いたい」というコマンドはこちらに配置するとよいでしょう。自分だけのワークフローを効率化するのに最適です。

### どちらを使うべきか

「プロジェクトコマンドと個人コマンド、どちらを使うべき？」と迷うこともあるかもしれません。

基本的には、チームで共有したいコマンドは**プロジェクトコマンド**、自分だけで使うコマンドは**個人コマンド**として管理するのがシンプルです。なお、同名のコマンドがある場合はプロジェクトコマンドが優先されます。これを活用すれば、チーム共有と個人カスタマイズを両立できるわけです。

## 基本的な作り方

カスタムコマンドの作成は簡単にできますので、以下の手順に従って作成してください。

前提としては `~/Desktop/command-test` というディレクトリに移動していることとします。

```
mkdir -p ~/Desktop/command-test
cd ~/Desktop/command-test
```

### 手順1: ディレクトリを作成

まず、コマンドを配置するディレクトリを作成してください。

```
# プロジェクトコマンドの場合
mkdir -p .claude/commands

# 個人コマンドの場合
mkdir -p ~/.claude/commands
```

### 手順2: Markdownファイルを作成

次に、コマンドの内容を記述したMarkdownファイルを作成しましょう。

以下の例では `echo` コマンドを使って `suggest-commit.md` ファイルを作成していますが、お好きな方法で作成してください。

```
# suggest-commit.md を作成
echo "mainブランチとの差分を確認し、適切なコミットメッセージを提案してください。" > .claude/commands/suggest-commit.md
```

これで `/suggest-commit` コマンドが使えるようになります。

引数を受け取るコマンドを作成したい場合は、以下の記事が参考になります。

[Claude Codeカスタムコマンドで$ARGUMENTSを使って引数を受け取る方法](https://zenn.dev/tmasuyama1114/books/claude_code_basic/viewer/custom-command-arguments)

### 手順3: コマンドを確認

最後に、作成したコマンドが認識されているか確認しましょう。

まず、Claude Codeを起動してください。

```
claude
```

`/` を入力すると、利用可能なコマンドがサジェストに表示されますので、その中に `/suggest-commit` が表示されていれば成功です。

```
───────────────────────────────────────────────────────────────────────────────────────────────────────────────
> /
───────────────────────────────────────────────────────────────────────────────────────────────────────────────
  /suggest-commit                             mainブランチとの差分を確認し、適切なコミットメッセージを提案し
                                              てください。 (project)
```

また、`/help` コマンドを実行した後、十字キーの右で `custom-commands` を選択すると、カスタムコマンドの一覧が表示されます。

```
> /help

──────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────
 Claude Code v2.0.69  general   commands   custom-commands   (tab to cycle)

 Browse custom commands:

 ❯ /suggest-commit                           mainブランチとの差分を確認し、適切なコミットメッセージを提案してください。 (project)
```

## 補足：サブディレクトリにコマンドを配置する方法

上記の例では `.claude/commands/` ディレクトリにコマンドを配置しましたが、サブディレクトリに配置することもできます。

```
$ mkdir .claude/commands/test
$ echo "helloとだけ出力してください" > .claude/commands/test/hello.md
```

以下のような構造で配置しています。

```
tree .claude/commands
.claude/commands
├── suggest-commit.md
└── test
    └── hello.md
```

この場合には `/test:hello` というコマンドが使えるようになります。

Claude Code を起動して候補を出すときにも `test:` というプレフィックスが表示されているので、関係するコマンドをまとめたい時などに便利です。

```
──────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────
> /test:hello
──────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────
  /test:hello     helloとだけ出力してください (project)
```

例えば Git 関連をまとめたり、テスト関連をまとめるなど、お好きな形でコマンドをまとめてください。

## コマンドの反映タイミング

新しいコマンドを追加した場合、すぐには反映されないため注意が必要です。

一度 Claude Code を終了して再起動すると、新しいコマンドが反映されるようになります。

`/clear` コマンドなど、Claude Codeの基本コマンドについては以下の記事で解説しています。

[Claude Codeの基本コマンド一覧：日常で使う主要コマンド](https://zenn.dev/tmasuyama1114/books/claude_code_basic/viewer/basic-commands)

```
# Claude Codeを終了
/exit

# 再度起動
claude
```

!

コマンドを**編集した場合も同様**にセッション再起動が必要です。編集しても反映されない場合は、まずこの手順を試してください。

## まとめ

この記事では、**カスタムコマンドの基本**を解説しました。

- `/コマンド名` でプロンプトを呼び出せる
- プロジェクト用と個人用の**2つの配置場所**がある
- コマンド追加後は `/clear` で反映

カスタムコマンドは「プロンプトのショートカット」という単純な仕組みですが、使い込むほど手放せなくなる機能です。まずは自分がよく使うプロンプトを1つコマンド化するところから始めてみてください。

# Claude Code Skillsとサブエージェントの使い分け：知識の提供とタスクの委譲

こんにちは、とまだです。

前回の記事では、Skillsの作成方法を解説しました。今回はいよいよ、多くの開発者が疑問に思うテーマに取り組みます。

!

Skillsの基本概念や作成方法については、以下の記事で詳しく解説しています。

[Claude CodeのSkillsとは？AIに知識を与える仕組みを解説](https://zenn.dev/tmasuyama1114/books/claude_code_basic/viewer/what-is-skills)

[Claude Code Skillsの作り方！SKILL.mdの書き方から references フォルダの活用まで](https://zenn.dev/tmasuyama1114/books/claude_code_basic/viewer/skills-creation)

「**Skills** と**サブエージェント**、結局どっちを使えばいいの？」

この疑問、とてもよく聞きます。どちらもClaude Codeの拡張機能ですし、似ているように見えますよね。「どっちも便利そうだけど、違いがよくわからない...」と感じている方も多いのではないでしょうか。

今回伝えたいことの要点は、「**知識を教えたいならSkills、仕事を任せたいならサブエージェント**」ということです。料理に例えるなら、「**レシピ本を渡すか、アシスタントに頼むか**」の違いになります。

この記事では、2つの機能の違いを明確にし、使い分けの判断基準から連携パターンまでを解説します。読み終わる頃には、迷わず適切な方を選べるようになるはずです。

## 忙しい人のために要約

この記事で押さえておきたいポイントは次の5つです。

- **Skills** = 「必要な時だけ開く参考書」
- **サブエージェント** = 「調査をお願いする別働隊」
- 「ルールを守らせたい」→ **Skills**
- 「調査や探索を任せたい」→ **サブエージェント**
- 並列実行時の競合に注意（書き込み系は要検討）

では、詳しく見ていきましょう。

## Skillsとサブエージェントの違い

まず、2つの機能を比較表で整理します。表だけではピンとこないかもしれませんが、後で具体例を交えて説明するので順番に見ていきましょう。

| 特徴 | Skills | サブエージェント |
| --- | --- | --- |
| **イメージ** | 参考書、マニュアル | アシスタント、別働隊 |
| **役割** | 知識の提供（How） | タスクの実行（Do） |
| **呼び出し方式** | モデル呼び出し（Claudeが自動判断） | Taskツールで明示的に起動 |
| **コンテキスト** | メインの会話に読み込まれる | **独立**（メインを汚さない） |
| **トークン消費** | 使う時だけ消費（節約効果あり） | 別の財布で消費（メインの節約になる） |
| **主な用途** | ルール適用、知識提供 | 調査、探索、検証 |
| **永続性** | タスクが終われば消える | タスクが終われば消える |

一見似ているように見えますが、役割が根本的に異なります。「どちらもClaudeの機能拡張なのに、なぜ分かれているの？」と思った方もいるかもしれません。ただ、それぞれに明確な得意分野があり、適切に使い分けることで開発効率が大きく変わってくると考えています。

### 料理で例えると

この違いを、料理の場面で考えてみましょう。

**Skills は「レシピ本」** のようなものです。

皆がさんが普段から料理をしているとしましょう。

レシピ本は棚にしまっておき、必要な時だけ取り出すものでしょう。「カルボナーラを作りたい」と思った時だけ、イタリアンのレシピ本を開いてそのページを参照する。これがSkillsの動きに相当します。

ここで大事なのは、**料理をするのは自分** という点です。レシピ本は「やり方」を教えてくれるものの、実際に手を動かすのは自分自身だからです。

一方で、**サブエージェント は「キッチンのアシスタント」** です。

アシスタントに「この前菜を作っておいて」と頼めば、材料の確認から調理まで全部やってくれます。  

その間、皆さんはメインディッシュに集中できるわけですね。

![](https://static.zenn.studio/user-upload/deployed-images/f4d03789d793b63b6564f63a.png?sha=0e4b2154b61e7ee016a8886e0974fe803c634e9f)

ここで大事なのは、サブエージェントの場合には**仕事を任せている** という点です。アシスタントがどれだけ大量の材料を使っても、皆さんの作業台が散らからないのは、作業場所が分離されているためです。（これがコンテキストの分離に相当します）

### 技術的な違い

ではここで、もう少し技術的な観点で整理してみましょう。

**Skillsの動作**:

1. Claudeが「この知識が必要」と判断する（**モデル呼び出し**）
2. SKILL.mdの内容がメインの会話に読み込まれる
3. その知識を使って、メインのClaudeが作業を続ける

**サブエージェントの動作**:

1. メインのClaudeが「このタスクを委譲しよう」と判断する
2. **Taskツールで明示的に起動**され、独立したコンテキストで別のClaudeが起動する
3. タスク完了後、結果だけがメインに報告される

つまり、Skillsは「自分の頭に知識を入れる」、サブエージェントは「別の人に仕事を頼む」という違いがあるわけです。

![](https://static.zenn.studio/user-upload/deployed-images/a2a82341d14101760c3de313.png?sha=389d1c5b2773eea2d61ddaa6732ec4574fbea588)

!

**呼び出し方式の違い**

**Skills** はモデル呼び出し方式で、Claudeがリクエストとdescriptionから自律的に判断します。  

**サブエージェント** はTaskツールで明示的に起動されます。

この違いは重要です。Skillsは「必要だと判断されれば自動で読み込まれる」のに対し、サブエージェントは「委譲すべきタスクがあると判断されたときに起動される」のです。ただ、どちらも「Claude が自律的に判断する」という点では共通しており、明示的にプロンプトで指示することで呼び出しの確実性を高めることもできます。

この区別がつくと、適切な選択ができるようになると考えています。なぜなら、「知識を使う」のと「仕事を任せる」のでは、求められる設計が全く異なるからです。

サブエージェントの基本的な仕組みについては、別の記事で詳しく解説しています。

[Claude Codeのサブエージェントとは？並列処理とコンテキスト分離の仕組み](https://zenn.dev/tmasuyama1114/books/claude_code_basic/viewer/what-is-subagent)

## 使い分けの判断チャート

実際に迷った時は、次の質問で判断できます。

**Q1**: AIに守らせたい**ルールや知識** ですか？

YESなら **Skills** を選びましょう。たとえば、デザインルールや命名規則、ライブラリのベストプラクティスなどが該当します。

**Q2** 大量のファイルを読んだり、**試行錯誤が必要な作業** ですか？

YESなら **サブエージェント** を選ぶとよいでしょう。たとえば、バグ調査、影響範囲の特定、ドキュメントの要約などがこれに該当します。

シンプルにまとめると、こうなります。

- 「**やり方**を教えたい」→**Skills**
- 「**調査や探索**を任せたい」→**サブエージェント**

ただし、「仕事を任せたい＝サブエージェント」と単純に考えるのは危険です。特に**書き込みや変更を伴う作業**をサブエージェントに任せる場合は注意が必要でしょう。なぜなら、複数のサブエージェントを並列実行すると、同じファイルを同時に編集しようとして**競合（コンフリクト）が発生する可能性**があるからです。

そのため、サブエージェントが最も活きるのは**読み取り中心の調査・探索タスク**になります。コードの調査、影響範囲の特定、ドキュメントの要約といった作業は、並列実行しても競合が起きにくく、サブエージェントの独立したコンテキストという特性を最大限に活かせるわけですね。

一方で、両方に当てはまるケースも存在します。たとえば「コードレビュー」は、ルールを適用する側面（Skills）と、大量のファイルを読む側面（サブエージェント）の両方を持っているからです。このような場合は後述する「連携パターン」が有効になります。

### よくある使い分け例

具体的なシーンで考えてみましょう。以下の表を見ると、判断基準がより明確になるかもしれません。

| シーン | 選択 | 理由 |
| --- | --- | --- |
| UIを作る時にデザインガイドに従わせたい | Skills | ルール・知識の提供 |
| プロジェクト全体でUserクラスの使用箇所を調査 | サブエージェント | 大量のファイル読み込み |
| コードレビュー時に命名規則をチェック | Skills | ルールの適用 |
| 新しいライブラリのドキュメントを読み込んで要約 | サブエージェント | 独立した調査タスク |
| テストコード作成時にテスト規約に従わせたい | Skills | 規約・知識の提供 |
| 既存の実装を客観的にテスト | サブエージェント | 独立した検証 |

## Skillsとサブエージェントを連携させる

ここまで「どちらを使うか」を見てきましたが、実は両者を組み合わせることも可能です。サブエージェントに専門知識を持たせるという使い方ですね。

ただし、ここで重要な仕様を押さえておく必要があります。**サブエージェントはデフォルトで親会話のSkillsを継承しません**。メインのClaudeで使えていたSkillsが、サブエージェントでは自動的に使えるわけではないのです。

そこで活用するのが、サブエージェントの定義ファイルにある `skills` フィールドです。必要なSkillsを明示的に指定することで、この制約を回避できます。

カスタムサブエージェントの作成方法については、別の記事で詳しく解説しています。

[Claude Codeのカスタムエージェント定義ファイルの作成方法](https://zenn.dev/tmasuyama1114/books/claude_code_basic/viewer/custom-agent-definition)

### skillsフィールドの使い方

サブエージェントの定義ファイルに `skills` フィールドを追加する方法を見てみましょう。

[https://code.claude.com/docs/en/sub-agents#file-format](https://code.claude.com/docs/en/sub-agents#file-format)

以下に例を示します。

```
---
name: migration-planner
description: データベーススキーマ変更を安全に計画
tools: Read, Write, Grep, Glob, Bash
model: sonnet
skills: migration-best-practices
---

データベースマイグレーションを安全に計画するエージェントです。
```

この例では、`migration-planner` というサブエージェントが、`migration-best-practices` というSkillを**起動時に自動的にロード**して参照しながら作業します。Skillsの知識を活かしつつ、独立したコンテキストで作業できるのがポイントです。

!

**skills フィールドの動作**

`skills` フィールドに指定されたSkillsは、サブエージェントの起動時に自動的にロードされます。これにより、メインのClaudeとは別のコンテキストで、専門知識を持ったサブエージェントが作業できるようになります。

なお、ビルトインエージェント（Explore、Plan、Verify）やTaskツール自体はSkillsにアクセスできません。カスタムサブエージェントのみが `skills` フィールドでSkillsを使用できます。

複数のSkillsを指定したい場合は、カンマ区切りで記述すればOKです。サブエージェントは指定されたすべてのSkillsを起動時にロードするようになります。

```
---
name: migration-planner
description: データベーススキーマ変更を安全に計画
tools: Read, Write, Grep, Glob, Bash
model: sonnet
skills: migration-best-practices, migration-best-practices-2
---
```

### 複数サブエージェントが異なるSkillsを使う

さらに応用として、複数のサブエージェントにそれぞれ異なるSkillsを持たせることも可能です。

```
サブエージェント A (プログラマー) → skills: ts-programming
サブエージェント B (テスター)     → skills: test-strategy
サブエージェント C (アーキテクト) → skills: ts-programming, aws-best-practices
```

各サブエージェントは独立したコンテキストで動作するため、必要なSkillsだけをロードできるわけですね。メインのコンテキストを圧迫することなく、専門知識を活用できるのが大きなメリットになります。

また、サブエージェントの並列実行にも対応しているため、複数の専門サブエージェントが同時に作業することも可能です。

「そんな複雑な構成、使いこなせるかな...」と思うかもしれませんが、最初はシンプルな構成から始めれば徐々に慣れていくかと思います。  

（私自身もまだ試行錯誤中ですが、シンプルな構成でも十分効果を感じています）

## 実践例

ここまでの内容を、実践的な例で確認してみましょう。

### 例1: コードレビューの自動化

コードレビュー時に、チームのコーディング規約を自動でチェックしたいというケースを考えてみましょう。

**設計**

```
[サブエージェント: code-reviewer]
       ↓ skills フィールドで参照
[Skill: team-coding-standards]
```

**サブエージェント定義** (`.claude/agents/code-reviewer.md`):

```
---
name: code-reviewer
description: PRの差分をレビューし、コーディング規約違反を指摘
tools: Read, Grep, Glob, Bash
model: sonnet
skills: team-coding-standards
---

プルリクエストの差分を読み込み、チームのコーディング規約に
違反している箇所を指摘するエージェントです。
```

**Skill定義** (`.claude/skills/team-coding-standards/SKILL.md`):

```
---
name: team-coding-standards
description: チームのコーディング規約とレビュー基準
---

# チームコーディング規約

変数名はcamelCase、定数はUPPER_SNAKE_CASE、クラス名はPascalCaseを使用します。
公開APIには必ずJSDocを記述し、複雑なロジックにはWhyコメントを追加します。
```

この構成では、code-reviewerがPRの差分を読み込みながら、team-coding-standardsの規約に照らし合わせてレビューを行います。人間が規約を覚えておく必要がなくなり、一貫したレビューが可能になります。

### 例2: ドキュメント更新ワークフロー

**目的**: コード変更に合わせて、関連ドキュメントを自動更新したい。

**設計**:

```
[メインのClaude]
       ↓ タスク委譲
[サブエージェント: doc-updater]
       ↓ skills フィールドで参照
[Skill: documentation-style]
```

この例では、メインのClaudeがコード変更を行った後、ドキュメント更新をサブエージェントに委譲します。doc-updaterはdocumentation-styleというSkillを参照しながら、チームのドキュメントスタイルに従って更新を行います。

## まとめ

Skillsとサブエージェントの使い分けをまとめておきます。

- **Skills** = 知識を提供（ルールを教えたい時）
- **サブエージェント** = タスクを委譲（仕事を任せたい時）
- `skills` フィールドで両者の連携も可能（ただし、サブエージェントはデフォルトでSkillsを継承しないので、明示的な指定が必要）

サブエージェントと Skills の連携は必須というわけではありませんが、サブエージェントにさらに専門知識を持たせることで、より高度なタスクを実行できるようになります。  

組み合わせられると「自分好み」にタスクを実行してくれますので、ぜひ試してみてください。

# Claude Codeのパーミッション設定を最適化する：セキュリティと利便性のバランス

こんにちは、とまだです。

「Claude Codeは便利だけど、勝手にファイルを消されたりしないか心配...」「毎回承認するのが面倒だけど、かといって全部許可するのも怖い...」と感じていませんか？

実際、AIエージェントが`rm -rf`を実行してディレクトリを消してしまったという悲劇は、海外のコミュニティでも定期的に話題になっています。

今回伝えたいことの要点は、**パーミッション設定を使えば「安全な操作は自動化、危険な操作はブロック」という理想的なバランスが実現できる**ということです。  

一言で言えば、設定次第で作業効率とセキュリティを両立できるわけです。

そこでこの記事では、パーミッションの基本概念から、3つのルール（allow、deny、ask）の使い分け、プロジェクト別の推奨設定、そして実際に検証してわかった注意点までを解説します。  

読み終わる頃には、自分のプロジェクトに最適なパーミッション設定を構築できるようになっているはずです。

## 忙しい人のために要約

この記事の要点を5つにまとめました。

- パーミッションは**操作を制御する設定**
- 3つのルール：`allow`、`deny`、`ask`
- **優先度はDeny > Ask > Allow**
- Bashは**ワイルドカード**（` *`）で柔軟にマッチ
- `/doctor`で**設定の問題を検出**できる

## パーミッションとは何か

パーミッションとは、**Claude Codeがどの操作を許可されるかを制御する設定**のことです。「セキュリティの門番」のようなものと考えるとわかりやすいでしょう。

というのも、Claude Codeはファイルの読み書き、シェルコマンドの実行、Webアクセスなど、豊富な機能を持っています。これは非常に便利ですよね。

ただ、この機能を適切に管理しないと、意図しないファイル削除や機密情報の漏洩といったリスクが生じる可能性があります。  

つまり、パーミッション設定は、こうしたリスクを軽減するためのセーフティネットとして機能するわけです。

「パーミッションを設定しなくてもClaude Codeは使えるのでは？」という疑問が生じるかもしれません。

確かに、デフォルトではすべての操作に対して確認プロンプトが表示されます。ですが、毎回承認するのは正直面倒ですよね。かといって、何も考えずに承認していると、危険な操作を誤って許可してしまう可能性も否定できません。パーミッション設定を適切に行えば、**安全な操作は自動承認し、危険な操作は確実にブロックする**という使い分けが可能になるのです。

## パーミッションの5つの種類

では、具体的にどのようなパーミッションがあるのでしょうか。Claude Codeのパーミッションは、以下の**5つのカテゴリ**に分類されます。

### 1. Bash（シェルコマンド）

まず、Bashパーミッションは、**ターミナルでのコマンド実行を制御する**ためのものです。おそらく最も重要なパーミッションといえるでしょう。

```
{
  "permissions": {
    "allow": ["Bash(npm run test *)"],
    "deny": ["Bash(rm -rf *)"]
  }
}
```

この例では、`npm run test`で始まるコマンドは許可しつつ、`rm -rf`のような危険なコマンドは禁止しています。

開発に必要なコマンドのみを許可するのがポイントです。

### 2. Read（ファイル読み取り）

次に、Readパーミッションは、**ファイルの読み取りを制御する**ためのものです。特に機密ファイルへのアクセスを制限したい場合に役立ちます。

```
{
  "permissions": {
    "allow": ["Read"],
    "deny": [
      "Read(.env)",
      "Read(.env.*)"
    ]
  }
}
```

基本的にはコードの読み取りは許可しつつ、環境変数ファイルのように機密情報を含むファイルは除外するのが良いでしょう。`.env`ファイルにはAPIキーやパスワードが含まれることが多いですから。

### 3. Edit（ファイル編集）

続いて、Editパーミッションは、**既存ファイルの編集を制御する**ためのものです。コード編集は開発の中心的な作業ですから、許可することが多い操作といえます。

```
{
  "permissions": {
    "allow": ["Edit"]
  }
}
```

ただ、設定ファイルや機密ファイルの編集を制限したい場合は、`deny`で特定のパターンを除外しておくと良いでしょう。

### 4. Write（ファイル作成）

また、Writeパーミッションは、**新規ファイルの作成を制御する**ためのものです。新しいコンポーネントやテストファイルを作成する際に使われます。

```
{
  "permissions": {
    "allow": ["Write"]
  }
}
```

ファイル作成を自動化したい場合は、このパーミッションを許可しておくと便利でしょう。

### 5. WebFetch（Webアクセス）

最後に、WebFetchパーミッションは、**外部URLへのアクセスを制御する**ためのものです。セキュリティを重視するプロジェクトでは、無効化を検討してみてもいいかもしれません。

```
{
  "permissions": {
    "deny": ["WebFetch"]
  }
}
```

外部通信が不要なプロジェクトでは、情報漏洩を防ぐために無効化しておくと安心です。

## 3つのルール：allow、deny、ask

パーミッション設定では、**3つのルール**を使って操作を制御できます。

### allow（許可）

`allow`に指定したパターンにマッチする操作は、**確認なしで自動実行**されます。

```
{
  "permissions": {
    "allow": [
      "Bash(npm *)",
      "Bash(git *)"
    ]
  }
}
```

開発中に頻繁に使うコマンドを登録しておくと、作業効率が大幅に向上します。

### deny（禁止）

`deny`に指定したパターンにマッチする操作は、**完全にブロック**されます。

```
{
  "permissions": {
    "deny": [
      "Bash(rm -rf *)",
      "Read(.env)",
      "Read(.env.*)"
    ]
  }
}
```

危険なコマンドや機密ファイルへのアクセスを防ぐために使います。

### ask（確認を強制）

`ask`に指定したパターンにマッチする操作は、`allow`**に含まれていても毎回確認プロンプトが表示**されます。

```
{
  "permissions": {
    "allow": ["Bash(git *)"],
    "ask": ["Bash(git push *)"]
  }
}
```

この例では、gitコマンドは基本的に自動実行されますが、`git push`だけは毎回確認が入ります。「普段は自動化したいけど、特定の操作だけは確認したい」という場面で便利です。

## ルールの優先度

3つのルールには**優先度**があります。これを理解しておかないと、意図した通りに動作しない場合があるので注意が必要です。

優先度は **Deny > Ask > Allow** の順番です。

つまり、同じ操作が複数のルールにマッチした場合、`deny`が最も強く、次に`ask`、最後に`allow`という順序で評価されます。

```
{
  "permissions": {
    "allow": ["Bash(echo *)"],
    "deny": ["Bash(echo secret *)"]
  }
}
```

この設定では、`echo hello`は自動実行されますが、`echo secret data`は拒否されます。`deny`が`allow`より優先されるからです。

一方で、`allow`と`ask`が競合した場合は`ask`が優先されるので、確認プロンプトが表示されます。

「基本はOKだけど、特定のケースだけ確認を入れたい」というニーズに応えられる設計になっているわけです。

## Bashパターンの書き方：ワイルドカード

ここで重要な注意点があります。Bashパーミッションのパターンは、**`*`（アスタリスク）を使ったワイルドカードマッチング**で動作します。

### 基本の書き方

ワイルドカードを使う場合は、スペースの後に`*`を付けます。

```
{
  "permissions": {
    "allow": [
      "Bash(npm run test *)",
      "Bash(git *)"
    ]
  }
}
```

この書き方で、「そのプレフィックスで始まるすべてのコマンド」にマッチするようになります。

### ワイルドカードの位置

`*`はコマンドの**任意の位置**に配置できます。末尾だけでなく、先頭や中間にも使えます。

```
{
  "permissions": {
    "allow": [
      "Bash(npm run *)",
      "Bash(* --version)",
      "Bash(git * main)"
    ]
  }
}
```

| パターン | マッチする例 |
| --- | --- |
| `Bash(npm run *)` | `npm run test`、`npm run build` など |
| `Bash(* --version)` | `node --version`、`npm --version` など |
| `Bash(git * main)` | `git checkout main`、`git merge main` など |

### スペースの有無による違い

`*`の前にスペースがあるかどうかで、マッチの挙動が変わります。

- `Bash(ls *)` → `ls -la` にマッチするが、`lsof` にはマッチ**しない**
- `Bash(ls*)` → `ls -la` にも `lsof` にもマッチする

スペースがある場合は**ワードバウンダリ**（単語の区切り）が適用されるため、意図しないコマンドにマッチするリスクが軽減されます。基本的にはスペースを入れる書き方を推奨します。

### 旧記法（`:*`）について

以前は`Bash(npm:*)`のようにコロン+アスタリスクで書く記法が使われていましたが、**現在は非推奨（deprecated）**です。`Bash(npm:*)` と `Bash(npm *)` は同じ動作をしますが、新しい記法を使いましょう。

### よくある間違い

以下のような書き方は正しく動作しません：

```
// ❌ 正規表現は使えない
"Bash(echo .*)"

// ❌ globの `**` は使えない
"Bash(rm -rf **)"
```

### Read/Editのパス記法

ReadやEditのパターンは以下のような記法で記述します。`.gitignore`の記法と同じです。

```
{
  "permissions": {
    "deny": [
      "Read(.env)",
      "Read(.env.*)",
      "Read(**/secrets/**)"
    ]
  }
}
```

パス指定には独自のルールがあります。

| 記法 | 意味 | 例 |
| --- | --- | --- |
| `//path` | ファイルシステムルートからの絶対パス | `Read(//Users/alice/secrets/**)` |
| `~/path` | ホームディレクトリからのパス | `Read(~/Documents/*.pdf)` |
| `./path` または `path` | 現在ディレクトリからの相対パス | `Read(.env)` |

## defaultModeの設定

`defaultMode`は、パーミッションで明示的に指定されていない操作に対する**デフォルトの動作を決定**するものです。

### 1. default（デフォルト）

まず、`default`は、**すべての操作に対して確認プロンプトを表示する**モードです。最も安全な選択肢といえるでしょう。

```
{
  "permissions": {
    "defaultMode": "default"
  }
}
```

毎回承認が必要になるので少し面倒に感じるかもしれませんが、意図しない操作を確実に防げます。セキュリティを最優先したい場合は、この設定がおすすめです。

### 2. acceptEdits

次に、`acceptEdits`は、**ファイルの読み取りと編集を自動承認する**モードです。

```
{
  "permissions": {
    "defaultMode": "acceptEdits"
  }
}
```

これを設定しておくと、毎回の起動後からすぐファイルの読み取りと編集が自動承認されるようになります。

### 3. bypassPermissions

これは `claude --dangerously-skip-permissions` フラグを使った場合のモードと同じです。

```
{
  "permissions": {
    "defaultMode": "bypassPermissions"
  }
}
```

危険なモードですので、必要な場合にのみ使用してください。

できるだけ `sandbox` モードを使用しておき、壊れても安全な環境で使うようにしましょう。

（`sandbox`モードの詳細は別な記事で解説します）

パーミッション設定と合わせて、Hooksを使った危険コマンドのブロックも検討してみてください。より強固なセキュリティ対策が実現できます。

## バランスの良い基本設定

まずは、多くのプロジェクトで使える汎用的な設定例を紹介します。

`rm -rf`の悲劇を防ぎつつ、開発効率も確保したバランス型の設定です。

```
{
  "permissions": {
    "allow": [
      "Bash(ls *)",
      "Bash(cat *)",
      "Bash(echo *)",
      "Bash(touch *)",
      "Bash(mkdir *)",
      "Bash(cp *)",
      "Read",
      "Edit",
      "Write"
    ],
    "deny": [
      "Bash(sudo *)",
      "Bash(rm -rf *)",
      "Bash(git reset *)",
      "Bash(git rebase *)",
      "Bash(wget *)",
      "Read(**/.env*)",
      "Read(id_rsa)",
      "Read(id_ed25519)",
      "Read(**/*token*)",
      "Read(**/*key*)",
      "Write(**/.env*)",
      "Write(**/secrets/**)"
    ],
    "ask": [
      "Bash(rm *)",
      "Bash(mv *)",
      "Bash(curl *)",
      "Bash(git add *)",
      "Bash(git commit *)",
      "Bash(git push *)",
      "Bash(git merge *)"
    ]
  }
}
```

この設定のポイントを説明します。

**denyで絶対に防ぐもの**

- `sudo` - root権限での実行は危険
- `rm -rf` - 強制削除は取り返しがつかない
- `git reset`/`git rebase` - 履歴の書き換えは慎重に
- `wget` - 外部からの無断ダウンロードを防止
- `.env`、SSH鍵、トークン系ファイル - 機密情報へのアクセスを遮断

**askで確認を挟むもの**

- `rm`/`mv` - ファイルの削除・移動は念のため確認
- `curl` - 外部通信は確認してから
- `git push`/`commit`/`merge` - リポジトリへの変更は確認

**allowで自動化するもの**

- `ls`、`cat`、`echo` - 参照系は安全
- `touch`、`mkdir`、`cp` - ファイル作成・コピーは頻繁に使う
- `Read`、`Edit`、`Write` - コード編集は開発の基本

この設定をベースに、プロジェクトの特性に合わせてカスタマイズしていくのがおすすめです。

## プロジェクト別の設定例

ここからは、プロジェクトの性質に応じた具体的な設定例を紹介します。

上記のベース例と組み合わせると、いろんなプロジェクトに対応できる設定が作れます。

自分のプロジェクトに近いものを参考にしてみてください。

### TypeScriptプロジェクト

TypeScriptプロジェクトでは、テストやビルド、型チェックなどを自動化しつつ、パッケージ操作やgitは確認を挟むのがおすすめです。

```
{
  "permissions": {
    "allow": [
      "Bash(npm run *)",
      "Bash(npm test *)",
      "Bash(npx tsc *)",
      "Bash(npx prettier *)",
      "Bash(npx eslint *)"
    ],
    "ask": [
      "Bash(npm install *)",
      "Bash(npm uninstall *)"
    ]
  }
}
```

`npm run`系は安全なので許可し、`npm install`/`uninstall`はパッケージを変更するので確認を挟みます。gitも意図しないコミットやpushを防ぐため`ask`に入れています。

### Pythonプロジェクト

Pythonプロジェクトでも同様に、スクリプト実行やテストは許可しつつ、パッケージ操作は確認を挟むのが安全です。

```
{
  "permissions": {
    "allow": [
      "Bash(python *)",
      "Bash(python3 *)",
      "Bash(pytest *)",
      "Bash(ruff *)",
      "Bash(mypy *)"
    ],
    "ask": [
      "Bash(pip install *)",
      "Bash(pip uninstall *)",
      "Bash(uv add *)",
      "Bash(uv remove *)"
    ]
  }
}
```

`pytest`やリンター系は許可しつつ、`pip install`/`uninstall`や`uv`のパッケージ操作は確認が必要です。

### 高セキュリティプロジェクト

機密性の高いプロジェクトでは、シェルコマンドを全面禁止し、ファイル操作のみを許可するという方法もあります。

```
{
  "permissions": {
    "allow": [
      "Read",
      "Edit"
    ],
    "deny": [
      "Bash(*)",
      "WebFetch"
    ]
  }
}
```

開発効率は下がりますが、セキュリティの面では最も安心できる設定です。

かなり厳しい設定とはなっているので、適宜許可するコマンドを増やしてください。

## 設定ファイルの配置場所

パーミッション設定は**複数の場所に配置でき**、それぞれスコープが異なります。「設定したのに反映されない」というトラブルを避けるためにも、優先順位を理解しておきましょう。

### 設定ファイルの優先順位

Claude Codeは以下の順序で設定を読み込みます。上位の設定が優先されるので、覚えておくと便利です。

1. **ローカル設定** - `.claude/settings.local.json`
2. **プロジェクト設定** - `.claude/settings.json`
3. **ユーザー設定** - `~/.claude/settings.json`

ローカル設定は個人用でGitに含めず、プロジェクト設定はチームで共有、ユーザー設定は全プロジェクト共通で適用されます。

### チーム開発での使い分け

チームで開発する場合は、以下のような使い分けが効果的です。

```
# プロジェクト共通設定（Gitに含める）
.claude/settings.json

# 個人設定（.gitignoreに追加）
.claude/settings.local.json
```

プロジェクト設定にはチーム共通のルールを記述し、個人設定には各メンバーの好みや追加の許可設定を記述するという使い分けがおすすめです。

## additionalDirectoriesの活用

`additionalDirectories`は、**プロジェクトディレクトリ外のファイルにアクセスを許可する**ための設定です。モノレポ構成や共有ライブラリがある場合に便利です。

```
{
  "permissions": {
    "additionalDirectories": [
      "../shared-libs/",
      "~/dotfiles/"
    ]
  }
}
```

なお、追加で参照するディレクトリは `/add-dir` というコマンドで追加することもできます。

ただ、アクセス範囲が広がるので、必要最小限のディレクトリのみを指定するようにしましょう。不要なディレクトリを追加すると、セキュリティリスクが高まってしまうからです。

## 実際に試してみた

ここからは、パーミッション設定の動作を実際に検証した結果を紹介します。今回の記事では、この検証結果を基準にご説明しています。

!

以下は検証時点（2026年1月）での結果です。Claude Codeのアップデートにより挙動が変わる可能性があります。

### 想定通りに動作した項目

以下の項目は、期待通りの動作が確認できました。

| 項目 | 確認できた動作 |
| --- | --- |
| allow | 指定パターンにマッチする操作が自動実行された |
| ask | allowに含まれていても確認プロンプトが表示された |
| ワイルドカードマッチング | `Bash(echo *)`で各種echoコマンドがマッチした |
| 優先度 | Deny > Ask > Allow の順に評価された |
| /doctor | 不正な設定パターンを検出してくれた |

### 注意が必要な項目

一方で、以下の点については想定外の挙動が確認できました。

#### denyの制限

`Bash(cat *)`を`deny`に設定しても、Claudeは別の方法でファイルを読み取ることがありました。

具体的には、catコマンドをブロックしても、Readツールを使ったり、echoコマンドを使ってファイルの内容を取得するケースが確認できました。これはClaude Codeが複数のツールを持っているためです。

つまり、**特定のツールをdenyしても、別のツールで同じ目的を達成される可能性がある**ということになります。完全にブロックしたい場合は、関連するすべてのツールをdenyする必要があるかもしれません。

#### 正規表現パターンの挙動

`Bash(echo .*)`のような正規表現パターンを設定すると、予期せぬ動作が発生しました。

`/permissions`コマンドでルールを確認しても、該当のルールが表示されませんでした。

ただ、`deny`に設定したルールでも、コマンド自体は実行できてしまう場合があり、ブロックも許可も意図通りに機能しない状態でした。

動作が不安定なため、**正規表現は使わない方が無難**です。素直に` *`によるワイルドカードマッチングのみを使いましょう。

#### --dangerously-skip-permissionsフラグ

`--dangerously-skip-permissions`フラグを使用した場合の挙動を検証しました。

```
claude --dangerously-skip-permissions
```

これは、先ほどお伝えした `defaultMode` の `bypassPermissions` モードを使用した場合と同じです。

```
{
  "permissions": {
    "defaultMode": "bypassPermissions"
  }
}
```

このフラグは「すべての権限チェックをスキップする」という名前なので、denyルールも無視されると思いがちです。しかし実際には、**permissions設定が優先される**仕様になっています。

- `deny`に設定したルールは**有効なまま**
- `ask`に設定したルールは**スキップされる**

つまり、このフラグを使っても危険なコマンドをブロックする`deny`設定は守られます。これは安全面で理にかなった設計といえますね。

一方で、`ask`による確認はスキップされるため、確認を強制したい操作がある場合は`deny`に設定しておくのが確実です。

Hooksの終了コードを使えば、さらに高度な制御も可能です。終了コード2を返すことで、特定の条件下でツール実行を完全にブロックできます。

[Claude Code Hooksの終了コードによる制御：ツール実行のブロック](https://zenn.dev/tmasuyama1114/books/claude_code_basic/viewer/hooks-exit-codes)

## トラブルシューティング

パーミッション設定でよくある問題と解決法を紹介します。困ったときは参考にしてみてください。

### /doctorコマンドで設定を検証

`/doctor`コマンドを実行すると、設定ファイルの問題点を指摘してくれます。

```
# Claude Code内で実行
/doctor
```

たとえば、Bashパターンで正規表現を使ってしまった場合、以下のような警告が表示されます。

```
Invalid Settings
 /Users/username/.claude/settings.json
  └ permissions
    └ allow
      └ "Bash(echo
        └ *)": Use " *" for prefix matching, not just "*".
        Change to "Bash(echo *)" for prefix matching.
```

設定がうまく動作しない場合は、まず`/doctor`で問題がないか確認してみましょう。

また、`/permissions`コマンドで設定を確認することもできます。

```
# 現在の設定を確認
/permissions
```

表示例:

```
> /permissions
─────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────
 Permissions: Allow   Ask   Deny   Workspace  (tab to cycle)

 Claude Code won't ask before using allowed tools.
 ╭─────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────╮
 │ ⌕ Search…                                                                                                                                                                               │
 ╰─────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────╯

 ❯ 1.  Add a new rule…
   2.  Bash(cat *)
```

最初に `allow` に設定したルールが表示され、Deny や Ask のルールは十字キーの「左右」で移動して確認できます。

もしパーミッションの挙動が想定外の場合は、`/permissions`コマンドで設定を確認してみましょう。

### 設定変更が反映されない

設定ファイルを変更した後は、**Claude Codeを再起動する必要があります**。これは意外と見落としやすいポイントです。

```
# セッションを終了して再起動
exit
claude
```

Claude Codeのセッション管理について詳しく知りたい場合は、こちらの記事も参考にしてください。

[Claude Codeのセッション管理：開始・終了・再開の完全ガイド](https://zenn.dev/tmasuyama1114/books/claude_code_basic/viewer/session-management)

## まとめ

パーミッション設定の要点を振り返っておきましょう。

- 5種類：Bash、Read、Edit、Write、WebFetch
- 3つのルール：`allow`、`deny`、`ask`
- **優先度はDeny > Ask > Allow**
- Bashは**ワイルドカード**（` *`）で柔軟にマッチ
- `/doctor`で設定を検証可能

一度設定すれば毎回の承認から解放されつつ、危険な操作はブロックされるようになります。ぜひ試してみてください。

パーミッション設定を含むClaude Code全体のベストプラクティスは、Anthropic公式が公開しているガイドで詳しく解説されています。

[Anthropicが公開しているClaude Codeのベストプラクティスを初心者向けに解説](https://zenn.dev/tmasuyama1114/articles/claude_code_best_practice_guide)

# Claude Codeのコンテキスト管理完全ガイド：/clear・/compact・/contextの使い分け

こんにちは、とまだです。

「Claude Codeで作業していたら、なんだか前の会話に引っ張られて変な回答が返ってくる...」と感じたことはありませんか？

また、少しずつ性能が落ちてきていると感じたことはありませんか？

今回伝えたいことの要点は、**コンテキスト管理を適切に行うことで、Claudeの判断精度とパフォーマンスを維持できる**ということです。一言で言えば、`/clear`・`/compact`・`/context`の3つのコマンドを状況に応じて使い分けることが重要です。

この記事では、コンテキストの仕組みから各コマンドの使い方、そして効果的な管理方法までを解説します。読み終わる頃には、タスク切り替え時のベストプラクティスが身についているはずです。

## 忙しい人のために要約

時間がない方は、まずこちらのポイントを確認してください。

- `/context`で使用量を確認し、高い場合は整理する
- `/clear`はタスク切り替え時に会話をリセット
- `/compact`は情報を残しつつトークンを節約
- コンテキスト汚染は判断精度・速度・コストに悪影響
- プロジェクト設定は`/clear`後も自動再読み込み

## コンテキストウィンドウとは

まず、**コンテキストウィンドウ**について理解しておきましょう。これはClaudeが一度に処理できるテキスト量の上限のことです。ここには現在の会話履歴だけでなく、さまざまな情報が含まれているわけです。

具体的には、以下の要素がコンテキストを消費します。

- **会話履歴**: やり取りの全記録
- **読み込んだファイル**: `@`記法で参照したコード
- **システムプロンプト**: CLAUDE.mdや設定ファイル
- **MCPからの情報**: 外部連携データ

これらすべてがコンテキストウィンドウ内に収まっている必要があるからです。上限を超えると、古い情報から自動的に削除されてしまいます。意図しない情報が消えると困るので、能動的に管理することが大切なのです。

### 確認すべきタイミング

以下のタイミングで`/context`を確認する習慣をつけると効果的です。

- 応答が遅くなったと感じたとき
- 回答の精度が落ちたと感じたとき
- 長時間のセッションを続けているとき
- 大きなファイルを読み込んだ後

## /clearコマンド：完全リセット

`/clear`を実行すると、現在のセッションの会話履歴がすべて削除されます。「コンテキストを空にして、不要な情報を一掃したい」というときに使うコマンドなのです。

Claude Codeの基本コマンドについてより詳しく知りたい方は、以下の記事も参考にしてください。

[Claude Codeの基本コマンド一覧：日常で使う主要コマンド](https://zenn.dev/tmasuyama1114/books/claude_code_basic/viewer/basic-commands)

```
/clear
```

実行後、コンテキストは完全に空の状態に戻ります。ただし、CLAUDE.mdや`.claude/rules/`などのプロジェクト設定は自動的に再読み込みされるため、プロジェクトのルールや規約は維持されます。

### /clearを使うべき4つのタイミング

`/clear`は、以下のような場面で使うと効果的です。

**1. タスクが完了し、別のタスクに移るとき**

機能Aの実装が終わり、まったく関係のない機能Bに取りかかる場合は、`/clear`でコンテキストをリセットするのがおすすめです。

```
[機能Aの実装完了]
> /clear
[新しいタスク「機能B」の開発を開始]
```

機能Aの情報が残っていると、Claudeが無関係な文脈を参照して混乱してしまうことがあるからです。

**2. 話題を大きく変えたいとき**

「ところで別の機能についてですが...」と話題を変えたい場面では、`/clear`が有効です。前の会話の文脈が残っていると、Claudeの応答が意図しない方向に引っ張られてしまうことがあります。

**3. 会話を引き継がなくても良い場合**

例えば、何か機能を実装した後、その機能に関するファイルにリファクタリング（整理）を行う場合は、前の会話を引き継がなくても良いでしょう。むしろ、新しい会話の方がフレッシュな目線でリファクタリングを行えるはずです。

**4. コンテキストが混乱している場合**

Claudeの回答が矛盾していたり、過去の情報に引きずられている場合は、一度`/clear`でリセットするのが効果的です。

会話の巻き戻しについては、`/rewind`コマンドも便利です。詳しくは以下の記事で解説しています。

[Claude Codeの/rewindコマンド完全解説：AIが壊したコードを一発復旧](https://zenn.dev/tmasuyama1114/articles/claude_code_checkpoint_rewind)

## /compactコマンド：要約して圧縮

`/compact`コマンドを実行すると、過去の会話を要約してトークン数を削減できます。ここで重要なのは、**会話内容が完全に消えるわけではない**という点です。要約された形で残るため、文脈を維持したまま会話を続けられるわけです。

```
/compact
```

この処理によってコンテキストウィンドウの空きが増えるので、新しい情報を追加できるようになるのです。

### トピックを指定した圧縮

`/compact`には引数を渡すことで、特定のトピックを重視した要約を指示できます。

```
/compact APIの仕様についてフォーカスして
```

このように指定すると、APIに関する情報を重点的に残しながら、それ以外の会話部分を圧縮してくれるわけです。複数の話題を扱った長いセッションで「この部分だけは残しておきたい」という場合に役立ちます。

| 場面 | コマンド例 |
| --- | --- |
| 設計議論を保持したい | `/compact アーキテクチャの決定事項にフォーカスして` |
| コードの変更履歴を残したい | `/compact 実装した機能の詳細にフォーカスして` |
| エラー解決の経緯を残したい | `/compact バグの原因と修正内容にフォーカスして` |

### 自動コンパクト機能

手動で`/compact`を実行しなくても、コンテキストウィンドウが満杯に近づくと自動的にコンパクトが実行される仕組みになっています。これはClaude Codeがコンテキスト使用量を監視しているからです。

ただし、自動コンパクトはClaude Codeの判断で実行されます。特定のトピックを優先するといった細かな制御はできないので、重要な情報を確実に残したい場合は手動で`/compact`を実行するのがおすすめです。

## /clearと/compactの使い分け

「`/clear`と`/compact`はどう違うの？」という疑問を持つ方も多いでしょう。この2つのコマンドには**異なる役割**があり、状況に応じて使い分けることが重要です。

### 基本的な違い

ポイントは**情報が残るかどうか**という点です。

| コマンド | 動作 | 情報の保持 |
| --- | --- | --- |
| `/clear` | 完全にリセット | 何も残らない |
| `/compact` | 要約して圧縮 | 重要な情報は残る |

チーム開発に例えてみましょう。`/clear`は、開発者に全く新しいタスクを指示するときに使うものです。一方で`/compact`は、誰かが進めてきたタスクの「引き継ぎ書」を作って、また別な誰かが続きを行うようなイメージです。この「引き継ぎ書」が、圧縮されたコンテキストに相当するわけです。

### 使い分けの判断基準

判断基準はシンプルなので、迷ったときは以下を参考にしてください。

| 状況 | 推奨コマンド |
| --- | --- |
| タスクが完全に終わった | `/clear` |
| 同じタスクを継続したい | `/compact` |
| コンテキストが混乱している | `/clear` |
| 重要な情報は残したい | `/compact` |

!

`/clear`を実行すると、これまでの会話内容は完全に消えます。重要な情報がある場合は、事前にドキュメントとして残しておくか、`/compact`を検討してください。

## コンテキスト汚染の影響

「毎回`/clear`するのは面倒では？」と思った方もいるかもしれません。

しかし、`/clear`を適切なタイミングで使わないと、**コンテキスト汚染**という問題が発生します。これは過去の会話が現在のタスクに悪影響を与える現象のことです。開発効率を低下させる原因になるので、ぜひ理解しておいてください。

| 問題 | 具体例 |
| --- | --- |
| **判断の混乱** | 過去のタスクの情報を参照し、現在のタスクに無関係な提案をする |
| **トークン消費の増加** | 不要な会話履歴がコンテキストを圧迫し、コストが増える |
| **応答速度の低下** | 処理すべき情報量が増え、応答が遅くなる |
| **矛盾した回答** | 過去の回答と現在の回答が矛盾する |

これらの問題を避けるためにも、タスクの切り替え時には`/clear`を使うことが重要です。

# CLAUDE.mdの@記法で他ファイルを参照する：仕様書やスタイルガイドの取り込み

「CLAUDE.mdに全部書こうとしたら、ファイルが長くなりすぎて困る...」と感じていませんか？

今回伝えたいことの要点は、**@記法を使えばCLAUDE.mdを分割管理できる**ということです。

一言で言えば、詳細な仕様書やスタイルガイドは別ファイルに分離し、CLAUDE.md本体はシンプルに保てるわけです。

この記事では、@記法の基本的な使い方から実践的な活用例まで解説していきます。

## 忙しい人のために要約

この記事のポイントを4つにまとめました。

- `@path/to/file.md`でファイル内容がコンテキストに読み込まれる
- 仕様書やスタイルガイドを別ファイルに分離できる
- パスはプロジェクトルートからの相対パスで記述する
- バッククォートで囲むと評価されないので注意

## @記法とは

@記法は、**CLAUDE.md内から他のファイルを参照するための仕組み**です。

CLAUDE.mdの基本的な概念については、別な記事で詳しく解説しています。

[CLAUDE.mdとは？Claude Codeのプロジェクト設定ファイルを理解する](https://zenn.dev/tmasuyama1114/books/claude_code_basic/viewer/what-is-claude-md)

例えば、API仕様書については `docs/api-specification.md` に分離したとしましょう。

この場合、CLAUDE.mdに `@docs/API_SPECIFICATION.md` と記述すると、そのファイルの内容がClaude Codeのコンテキストに自動的に読み込まれます。

これにより、CLAUDE.md本体には概要や重要なルールだけを書き、詳細な情報は専用ファイルに分離できるわけです。

つまり、「CLAUDE.mdが長くなりすぎて読みにくい」という問題を解決するための機能といえるでしょう。CLAUDE.mdを書いていて「もう少しスッキリさせたいな」と感じたことがある方には、ぜひ試してほしい機能です。

## 基本的な使い方

@記法の書式はとてもシンプルです。「これなら自分でもできそう」と思っていただけるのではないでしょうか。

```
## API仕様
@docs/API_SPECIFICATION.md

## コーディングスタイル
@docs/CODING_STYLE.md

## データベーススキーマ
@docs/DATABASE_SCHEMA.md
```

上記のように記述すると、Claude Codeは起動時にこれらのファイルを読み込み、プロジェクトのコンテキストとして認識してくれます。つまり、Claudeが作業を始める時点で、これらのドキュメントの内容を把握した状態になるということです。

### パスの書き方

パスはプロジェクトルートからの相対パスで記述します。「どこを基準にすればいいの？」と迷う方もいるかもしれませんが、プロジェクトのルートディレクトリ（CLAUDE.mdがある場所）からの相対パスと覚えておけばOKです。

| 記述例 | 参照先 |
| --- | --- |
| `@docs/spec.md` | プロジェクトルートの `docs/spec.md` |
| `@src/types/README.md` | `src/types/README.md` |
| `@CONTRIBUTING.md` | プロジェクトルートの `CONTRIBUTING.md` |

## 複数ファイルの参照

複数のファイルを参照する場合は、それぞれの@記法を並べて記述するだけです。「たくさんのドキュメントがあるけど、どう整理すればいい？」という疑問があるかもしれません。カテゴリ別に整理して記述すると、後から見たときに分かりやすくなります。

```
## プロジェクト固有の情報

プロジェクトに必要な情報を分野別に参照します。

### API関連
@docs/api/endpoints.md
@docs/api/authentication.md

### 設計ドキュメント
@docs/architecture/overview.md
@docs/architecture/database.md

### 開発ガイドライン
@docs/guidelines/code-style.md
@docs/guidelines/testing.md
```

参照先のファイルはすべてClaude Codeのコンテキストに読み込まれます。ただし、参照するファイルが多すぎるとコンテキストを消費してしまうため、本当に必要なファイルに絞ることをおすすめします。

## コードスパンで囲むと評価されない

@記法を使う際に、初心者がつまずきやすいポイントがあります。

**バッククォート（コードスパン）で囲むと、@記法が評価されません**。この違いに気をつけてください。

```
❌ `@docs/spec.md`     → これは評価されない（ただの文字列）
✅ @docs/spec.md       → これは評価される（ファイルが読み込まれる）
```

この挙動は、Markdown内で@記法の例を示したい場合には便利な特徴といえます。実際にファイルを読み込ませたくない場合は、意図的にバッククォートで囲むとよいでしょう。

一方で、うっかりバッククォートで囲んでしまうと「ファイルが読み込まれない」という状態になってしまいます。「設定したはずなのに参照が機能しない...」という場合は、まずバッククォートで囲まれていないか確認してみてください。

## 実践的な活用例

「実際にはどう使えばいいの？」と思う方も多いのではないでしょうか。

ここでは、**3つの具体例**を紹介します。それぞれ異なるユースケースを想定しているので、ご自身のプロジェクトに近いものを参考にしてください。

### 例1: API仕様書の分離

CLAUDE.mdにすべてのAPIエンドポイントを書くと、すぐに数百行になってしまいますよね。そんなときは、API仕様を別ファイルに分離するのがおすすめです。

```
# MyProject

## 概要
タスク管理アプリケーション

## API仕様
詳細なAPI仕様は以下を参照してください。
@docs/api-reference.md

## コーディング規約
以下のルールに従ってください。

- TypeScript strict モード必須
- 関数は50行以内
```

この例では、`docs/api-reference.md` にエンドポイントの詳細な定義、リクエスト/レスポンスの形式、エラーコードなどを記述しています。このように分離することで、CLAUDE.md本体がすっきりした状態を保てるのです。

### 例2: 複数チーム向けの構成

フロントエンドとバックエンドで異なるガイドラインがある場合にも、@記法は有効です。以下の例では、各チームのガイドラインを別ファイルで管理しています。

```
# MyProject

## チーム共通ルール
以下は全チームで守るべきルールです。

- コミットメッセージは Conventional Commits 形式
- PR は必ずレビューを受ける

## フロントエンド
@docs/frontend/guidelines.md

## バックエンド
@docs/backend/guidelines.md

## インフラ
@docs/infrastructure/guidelines.md
```

各チームは自分たちのガイドラインを独立して管理できるようになります。さらに、CLAUDE.md本体はシンプルに保てるというメリットもあるのです。チーム規模が大きくなるほど、この分離のありがたみを実感できるでしょう。

### 例3: 外部仕様書との連携

「既存のドキュメントがあるのに、わざわざCLAUDE.md用に書き直すのは面倒...」と思いませんか？ @記法なら、既存のドキュメントをそのまま活用できます。

```
# MyProject

## プロジェクト概要
社内向け経費精算システム

## 業務仕様
@docs/business-requirements.md

## データベース設計
@docs/database-schema.md

## 外部API連携
@docs/external-api-integration.md
```

プロジェクトに既存のドキュメントがある場合、それを@記法で参照するだけでClaude Codeのコンテキストに含められます。新たにCLAUDE.md用のドキュメントを作り直す必要がないのは嬉しいポイントです。

## @記法と.claude/rules/の使い分け

「@記法と.claude/rules/ディレクトリの違いは何？」という疑問が生じることがあります。

両者には**明確な違い**があるので、以下の表で整理しましょう。

| 項目 | @記法 | .claude/rules/ |
| --- | --- | --- |
| 配置場所 | 任意（プロジェクト内のどこでも） | `.claude/rules/`ディレクトリ固定 |
| 参照方法 | CLAUDE.md内で明示的に記述 | 自動的に読み込まれる |
| 条件付き適用 | なし | pathsフロントマターで可能 |
| 用途 | 既存ドキュメントの参照 | プロジェクトルール |

使い分けの指針をまとめると、こうなります。

**@記法**は、既存のドキュメントをそのまま参照したい場合に向いています。API仕様書や設計書など、すでにあるドキュメントを活用できるのが魅力です。

一方で、 `.claude/rules/` は、Claude Code専用のルールを条件付きで適用したい場合に適しています。特定のファイルパターンに対してだけルールを適用したい、といったケースで便利です。

## 注意点とベストプラクティス

@記法は便利な機能ですが、いくつか押さえておきたいポイントがあります。

### コンテキストの消費に注意

参照先ファイルが大きいと、その分だけコンテキストを消費してしまいます。  

不要なファイルを参照すると、実際の作業に使えるコンテキストが減ってしまうのです。

本当に必要なファイルだけを参照するようにしましょう。「このドキュメントは本当に必要か？」と自問してみることをおすすめします。

コンテキスト管理の詳細については、別な記事で解説しています。

[Claude Codeのコンテキスト管理完全ガイド：/clear・/compact・/contextの使い分け](https://zenn.dev/tmasuyama1114/books/claude_code_basic/viewer/context-management)

### 相対パスの起点

パスの起点は**常にプロジェクトルート**になります。

```
❌ ../docs/spec.md        → 相対パスは使えない
✅ docs/spec.md           → プロジェクトルートからのパス
```

たとえば以下のようなファイル構造で `CLAUDE.md` と `docs/spec.md` があるとしましょう。

```
my-project/
├── CLAUDE.md
├── docs/
│   └── spec.md
```

この場合、 `CLAUDE.md` から `docs/spec.md` を参照する場合は、以下のように記述します。

```
@docs/spec.md
```

CLAUDE.mdの配置場所と優先順位については、別な記事で詳しく解説しています。

[CLAUDE.mdの配置場所と優先順位：チーム共有と個人設定の両立](https://zenn.dev/tmasuyama1114/books/claude_code_basic/viewer/claude-md-hierarchy)

### ファイルの更新反映

参照先ファイルを更新した場合、**次回のClaude Code起動時**に反映されます。

「更新したのに反映されない...」という場合は、現在のセッションを一度終了してから再度起動してみてください。

- @記法で外部ファイルをCLAUDE.mdから参照できる

CLAUDE.mdが長くなりすぎたと感じたら、ぜひ@記法を試してみてください。プロジェクトの可読性と保守性がぐっと向上します。

# Claude Codeのサブエージェントとは？並列処理とコンテキスト分離の仕組み

「Claude Codeで大規模なコードベースを調査すると、コンテキストがすぐにいっぱいになってしまう...」と感じたことはありませんか？

今回伝えたいことの要点は、**サブエージェントを使えばコンテキストを節約しながら複雑なタスクを効率的に処理できる**ということです。

この記事では、サブエージェントの基本概念から組み込みの3種類、実践的なユースケースまでを解説します。読み終わる頃には、大規模なコードベース調査や複数ファイルの同時編集を効率的に進められるようになっているはずです。

## 忙しい人のために要約

この記事のポイントを5つにまとめました。

- サブエージェントは**独立したコンテキスト**で動作する専門エージェント
- 複雑なタスクを分割して**並列処理**できる
- メインセッションのコンテキストを消費せずに**深い調査が可能**
- 組み込みで**5種類のエージェント**を用意（執筆時点）
- `.claude/agents/`に配置して**カスタムサブエージェント**も作成可能

## サブエージェントとは

サブエージェントは、Claude Codeのメインエージェントが特定のタスクを委任するための「分身」です。メインのClaudeとは別のコンテキストウィンドウで動作するため、専門的なタスクを独立して処理できるわけです。

チームで仕事をするのに似ています。

メインエージェント（普通に会話するClaude）がプロジェクトマネージャーで、サブエージェントが営業チームとしましょう。

プロジェクトマネージャー（メインエージェント）が営業チーム（サブエージェント）に「来月のクライアント資料を準備して」と伝えれば、営業チームが自分たちで素材集めから資料作成までやってくれますよね。

細かい指示を出さずに結果だけを待つ感じです。

具体的には、以下の流れで動作します。

```
> このコードベースのエラーハンドリングを調査して

=> メインエージェント: 「サブエージェントに調査を依頼します」
=> サブエージェント: (独立して調査 → 結果をまとめて報告)
```

ここで押さえておきたいのが、**コンテキストの独立性**というサブエージェントの最大の特徴です。

言い換えれば、サブエージェントは自分専用のコンテキストウィンドウを持っているため、調査の途中経過や試行錯誤がメインセッションに影響することはありません。

最終的な結果だけがメインエージェントに返されるからこそ、コンテキストを効率的に使えるのです。

## なぜサブエージェントが必要なのか

サブエージェントが必要とされる理由は、大きく3つあります。

### 1. 複雑なタスクの分割と並列処理

大規模なコードベースで「すべてのAPIエンドポイントをドキュメント化して」といった広範な作業を依頼する場面を想像してみてください。1つのエージェントですべてを処理すると、どうしても時間がかかりますよね。

これは、1人で大量の書類をすべて整理しようとするのと同じです。一方で、複数の同僚に異なる書類の束を渡して並列で整理させれば、圧倒的に速く終わりますよね。サブエージェントを使えば複数のモジュールを並列で処理できるため、大幅に時間を短縮できます。

### 2. コンテキストの効率的な管理

これが最も大きなメリットかもしれません。

というのも、Claude Codeには利用可能なコンテキストウィンドウに上限があるため、深い調査や大量のファイル読み込みを行うとメインセッションのコンテキストがすぐに埋まってしまいます。

例えるなら、頭の中に入る情報量に限りがあるようなものです。

デスクに積まれた書類すべてを机の上に広げて読むのではなく、他人に読ませて要点だけを報告させれば、自分の机はすっきり保ったまま仕事が進みます。

サブエージェントは独自のコンテキストを持つため、深い調査を行っても問題ありません。メインセッションには結果のサマリーだけが返されるので、コンテキストを効率的に使い続けられるわけです。

### 3. 専門性の発揮

特定のタスクに特化したプロンプトを持つサブエージェントを作成すると、より高品質な結果が得られるようになります。

例えば、コードレビュー専門、テスト作成専門、デバッグ専門など、役割を分けることでそれぞれが得意な分野に専念できるのです。

これは、飲食店で調理師、ホール担当、事務担当が各自の役割に集中するのと似ています。それぞれが専門分野で力を発揮することで、全体のクオリティが上がるわけです。

!

**コンテキスト管理のポイント**

サブエージェントを使う最大のメリットは、メインセッションのコンテキストを圧迫しないことです。「このプロジェクトの認証周りを調べて」と依頼した場合、サブエージェントは数十ファイルを読み込んで調査しますが、メインセッションに返されるのは調査結果のサマリーだけ。この違いは大きいですよね。

## Claude Code組み込みのサブエージェント

Claude Codeには、最初から利用可能な5種類の組み込みエージェント（＝最初から利用可能なサブエージェント）が用意されています。

先ほど `/agents` コマンドで表示した結果のうち、`Built-in agents (always available)` という項目に表示されているエージェントが組み込みエージェントです。

それぞれが異なる専門分野を持っているため、タスクの性質に応じて最適なエージェントが自動選択されます。

| 名前 | モデル | 用途 |
| --- | --- | --- |
| **general-purpose** | Sonnet | 複雑な複数段階のタスク、探索と修正の両方 |
| **Explore** | Haiku | 高速な読み取り専用のコードベース検索 |
| **Plan** | inherit | 計画モード中のコードベース調査 |
| **claude-code-guide** | Haiku | Claude Codeおよびエージェント SDK の公式ドキュメント参照 |
| **statusline-setup** | Sonnet | ステータスラインの設定 |

これらは自動的に選択され、適切な場面で実行されるので、特に意識する必要はありません。

ただ、存在を知っておくと便利な場合もありますので、それぞれの特徴について簡単に解説します。

### general-purposeエージェント

general-purposeエージェントは、最も汎用的なサブエージェントです。そのため、調査だけでなくファイルの編集やコマンドの実行も可能で、複雑な複数段階のタスクを自律的に処理できます。ただし、権限が広い分、意図しない変更には注意が必要でしょう。

### Exploreエージェント

Exploreエージェントは、コードベースの調査に特化した読み取り専用のサブエージェントです。しかも、Haikuモデルを使用しているため、高速かつ低コストで動作するのが魅力です。

```
「このプロジェクトでRedisを使っている箇所を探して」
→ Exploreエージェントが自動的に起動して調査
```

ファイルを編集する権限は持たないため、安全に広範な調査を任せられます。ただし、調査結果に基づいて編集を行いたい場合は、別のエージェントを使う必要がある点には注意してください。

### Planエージェント

Planエージェントは、計画モード（`/plan`コマンドで開始）で使用されるエージェントです。タスクの実装計画を立てる際に、コードベースを調査して必要な情報を収集してくれます。

### claude-code-guideエージェント

claude-code-guideエージェントは、Claude Codeおよびエージェント SDK の公式ドキュメントを参照するための専門エージェントです。Claude Codeの機能、使い方、設定方法、SDK の使用方法に関する質問に対して、最新の公式情報を基に回答してくれます。

これがあるおかげで、Claude Code は Claude Code 自身の機能を把握しながら、ユーザーの質問に答えることができます。

そのため、たとえば使えるコマンドの種類や、サブエージェントの設定方法についても、自分で調べなくてもすぐに答えてくれます。

### statusline-setupエージェント

statusline-setupエージェントは、Claude Codeのステータスラインをカスタマイズする際に使用されます。

ステータスラインというのは、Claude Code の下部に表示されるテキストです。

たとえば、Claude Code を動かしているカレントディレクトリ名などを表示することができます。

`/statusline`というコマンドでも設定が可能なのですが、少し高度な内容となりますので、通常は意識しなくて問題ありません。

詳細は以下の記事をご覧ください。

[https://dev.classmethod.jp/articles/claude-customize-status-line/](https://dev.classmethod.jp/articles/claude-customize-status-line/)

そして、`statusline-setup` エージェントはユーザーの設定を読み込んで、パーソナライズされたステータスラインの構成をサポートしてくれます。

## サブエージェントのユースケース

サブエージェントが効果を発揮する具体的なシナリオを紹介します。実務での活用例を思い浮かべながら、読んでみてください。

### 大規模なコードベース調査

数十から数百のファイルにまたがる調査も、サブエージェントなら効率的に処理できます。

```
「このプロジェクトで認証がどのように実装されているか調査して」
```

調査結果はサマリーとしてまとめられるため、メインセッションのコンテキストを圧迫しません。調査後も余裕を持って作業を続けられるのは嬉しいですよね。

### 複数ファイルの並列編集

複数のファイルに対する同様の変更を、並列で処理したい場面もあるでしょう。

```
「すべてのコントローラーにロギング処理を追加して」
```

ただし、並列で編集する場合は変更の競合に注意が必要です。同じファイルを複数のサブエージェントが同時に編集するのは避けましょう。

### ドキュメントの自動生成

さらに、複数のエンドポイントを並列で処理し、統一されたフォーマットでドキュメントを生成することもできます。

```
「すべてのAPIエンドポイントのドキュメントを生成して」
```

## サブエージェントとメインエージェントの関係

「サブエージェントとメインエージェントはどう連携しているのか？」という疑問が浮かぶかもしれません。ここで整理しておきましょう。

メインエージェントとサブエージェントの関係は、マネージャーと専門家チームの関係に似ています。以下の4段階で動作します。

1. **タスクの委任**: メインエージェントが「この調査をやって」とサブエージェントにタスクを渡す
2. **独立した実行**: サブエージェントが自分のコンテキストで自律的にタスクを実行
3. **結果の報告**: 完了したら、結果のサマリーをメインエージェントに返す
4. **統合と応答**: メインエージェントが結果を統合してユーザーに報告

この仕組みがあるからこそ、複雑なタスクでも効率的に処理できるわけです。

コンテキスト管理の詳細については、別の記事で解説しています。

[Claude Codeのコンテキスト管理完全ガイド：/clear・/compact・/contextの使い分け](https://zenn.dev/tmasuyama1114/books/claude_code_basic/viewer/context-management)

サブエージェントの呼び出しはどう判断される？

実は、Claude Codeはタスクの性質に応じて自動的にサブエージェントを呼び出すかどうかを判断しています。

- **広範な調査が必要な場合**: Exploreエージェントが自動選択される
- **複雑な複数段階のタスク**: general-purposeエージェントが選択される
- **計画モード中**: Planエージェントが使用される

明示的にサブエージェントを指定することも可能です。

```
「Exploreエージェントでこのプロジェクトの依存関係を調べて」
```

# CLAUDE.mdとは？Claude Codeのプロジェクト設定ファイルを理解する

こんにちは、とまだです。

「Claude Codeを使い始めたけど、毎回プロジェクトの説明をするのが面倒...」と感じていませんか？

今回伝えたいことの要点は、**CLAUDE.mdというファイル1つで、その手間がなくなる**ということです。

CLAUDE.mdは、プロジェクトの文脈をClaude Codeに伝えるための設定ファイルです。このファイルを配置しておくと、Claude Codeがプロジェクトの背景を理解した状態で作業を始めてくれるようになるんです。

この記事では、CLAUDE.mdの基本的な役割から、`/init`コマンドでの自動生成、効果的な書き方まで解説していきます。読み終わる頃には、すぐに自分のプロジェクトでCLAUDE.mdを活用できるようになっているはずです。

## 忙しい人のために要約

この記事の要点を5つにまとめました。

- **CLAUDE.md**はプロジェクトのルールや文脈をClaude Codeに伝える設定ファイル
- Markdown形式で記述し、**プロジェクトルートに配置**する
- `/init`**コマンド**でプロジェクトを分析して自動生成できる
- プロジェクト概要・技術スタック・コーディング規約などを記載する
- **グローバルルール**（`~/.claude/CLAUDE.md`）で全プロジェクト共通のルールも設定可能

## CLAUDE.mdとは

CLAUDE.mdは、Claude Codeがプロジェクトに関する情報を「記憶」するためのファイルです。プロジェクトの概要、使用技術、コーディング規約などを記述しておくと、セッションをまたいで情報を永続化できるんです。

RPGに例えると、CLAUDE.mdは「世界観」の設定に相当します。

「中世ファンタジー」「魔法が存在する」といった世界観は、プレイヤーがどのエリアにいても変わらない前提条件として機能しますよね。

同様に、CLAUDE.mdに書かれた内容は、Claude Codeがどのファイルを編集していても常に参照される基本設定として働くわけです。

!

CLAUDE.mdという名前は**大文字で記述する**のがポイントです。`claude.md`や`Claude.md`ではなく、`CLAUDE.md`と指定してください。ファイル名の大文字・小文字の違いで認識されないことがあるため、この点は覚えておくとよいでしょう。

## CLAUDE.mdがあると何が変わるか

CLAUDE.mdを配置すると、開発体験が大きく変わります。私自身、CLAUDE.mdを活用するようになってから「毎回同じ説明をする」というストレスがほぼなくなりました。

具体的なメリットをあげてみましょう。

- **コンテキストの共有**: プロジェクトの文脈を理解した状態で作業開始
- **説明の省略**: 毎回同じ説明をする手間がなくなる
- **規約の遵守**: コーディング規約を守らせやすくなる
- **チーム共有**: 全員が同じルールでClaude Codeを使える

ただし、CLAUDE.mdはあくまで「指示」であり、Claudeがすべてを厳密に従うわけではないことには注意が必要です。

複雑なルールや曖昧な表現は意図通りに解釈されない場合もあります。そのため、ルールは**具体的かつ簡潔**に記述することをおすすめします。

## /initコマンドで自動生成する

「CLAUDE.mdを一から書くのは大変そう...」と思うかもしれませんが、心配はいりません。

`/init`コマンドを使えば、Claude Codeがプロジェクトを分析して雛形を自動生成してくれます。

Claude Code を起動したら、以下のコマンドを実行してください。

```
# 初期化コマンドを実行
/init
```

`/init`を実行すると、Claude Codeは以下の4つの処理を順番に行います。

1. まず、プロジェクト構造を分析して、ファイルとディレクトリの構成、使用言語・フレームワークを検出する
2. 次に、README.md、package.json、requirements.txtなどの重要なファイルを読み取る
3. 続いて、プロジェクト概要、技術スタック、ディレクトリ構造、開発ガイドラインを含むCLAUDE.mdを生成する
4. 最後に、生成された内容の確認を求める

/initコマンドを含むClaude Codeの基本コマンドについては、別の記事で詳しく解説しています。

[Claude Codeの基本コマンド一覧：日常で使う主要コマンド](https://zenn.dev/tmasuyama1114/books/claude_code_basic/viewer/basic-commands)

生成された内容がすべて適切とは限らないこともあります。

しかし、ゼロから書くよりはるかに効率的なので、生成後に手動で調整していくのがおすすめです。

## CLAUDE.mdの基本構造

効果的なCLAUDE.mdを作成するには、必要な情報を適切なセクションに整理することが大切です。

「どんな項目を書けばいいの？」という疑問もあると思うので、私が実際に使っている構成例を紹介します。

```
# プロジェクト名

## 概要
プロジェクトの目的と主要機能の簡潔な説明を書きます。

## 技術スタック
使用している技術を一覧で記載します。
- Frontend: React 18 + TypeScript 5
- Backend: Node.js 20 + Express
- Database: PostgreSQL

## ディレクトリ構造
主要なディレクトリとその役割を記載します。
src/
├── components/  # Reactコンポーネント
├── hooks/       # カスタムフック
├── utils/       # ユーティリティ関数
└── types/       # TypeScript型定義

## コーディング規約
チームで守るべきルールを明記します。
- TypeScriptのstrictモードを使用してください
- コンポーネントはfunctional componentで記述してください
- 関数名はcamelCase、コンポーネント名はPascalCaseで書いてください

## 禁止事項
やってほしくないことを記載します。
- anyの使用は最小限にしてください
- console.logを本番コードに残さないでください
- 既存のテストを削除しないでください

## よく使うコマンド
開発で頻繁に使用するコマンドを記載します。
- npm run dev: 開発サーバー起動
- npm test: テスト実行
- npm run build: 本番ビルド
```

この構成はあくまで一例なので、プロジェクトの特性に応じてセクションを追加・削除してください。

!

CLAUDE.mdが長くなりすぎると、Claudeのコンテキストウィンドウを圧迫してしまいます。重要な情報に絞り、詳細なルールは`.claude/rules/`ディレクトリに分離することも検討してみてください。

CLAUDE.mdの肥大化を防ぐモジュール化の詳細は、別の記事で解説しています。

## 効果的な指示の書き方

CLAUDE.mdに記述する指示は、**具体的かつ簡潔であること**がポイントです。

曖昧な指示は意図通りに解釈されないこともあります。

色々と試すうちに好みの設定が見つかるかもしれませんが、まずは以下のパターンを参考にしてみてください。

### やってほしいこと（推奨事項）

推奨事項は「〜してください」「〜を使ってください」という形式で記述します。この形式が指示として最も明確に伝わりやすいはずです。

CLAUDE.mdとは異なる知識提供の仕組みとして、Skillsという機能もあります。詳しく知りたい方は以下の記事をご覧ください。

[Claude CodeのSkillsとは？AIに知識を与える仕組みを解説](https://zenn.dev/tmasuyama1114/books/claude_code_basic/viewer/what-is-skills)

例:

- エラーハンドリングを考慮して実装してください
- importはsrc/からの絶対パスを使用してください
- 実装前にテストコードを作成し、テストをパスさせるよう実装してください

### やってほしくないこと（禁止事項）

一方で、禁止事項は「〜しないでください」「〜は避けてください」という形式で記述します。

こういった表現はClaude Codeに対して明確な制約として機能します。

- 型定義としてany型を使用しないでください
- マジックナンバーを直接書くことは避けてください
- 既存のAPI仕様を勝手に変更しないでください

## グローバルルールファイルを使う

ここまでプロジェクトルートに配置するCLAUDE.mdを紹介してきましたが、実は**全プロジェクト共通で適用されるグローバルルール**も設定できるんです。

`~/.claude/CLAUDE.md`（ホームディレクトリ配下）にファイルを配置すると、どのプロジェクトでClaude Codeを起動しても、そのルールが自動的に読み込まれます。

### プロジェクト単位とグローバルの使い分け

両者の使い分けを整理すると、以下のようになります。

| 種類 | 配置場所 | 用途 |
| --- | --- | --- |
| プロジェクト単位 | プロジェクトルート | そのプロジェクト固有のルール（技術スタック、ディレクトリ構造など） |
| グローバル | `~/.claude/CLAUDE.md` | 全プロジェクト共通のルール（コード品質の基準、エラーハンドリング方針など） |

たとえば、「エラーを握りつぶさない」「セキュリティを意識する」「テストをスキップしない」といった普遍的なルールはグローバルに、「このプロジェクトではReact を使う」「コンポーネントはsrc/components/に配置する」といった固有のルールはプロジェクト単位で設定するのがおすすめです。

CLAUDE.mdの配置場所と優先順位の詳細については、別の記事で解説しています。

[CLAUDE.mdの配置場所と優先順位：チーム共有と個人設定の両立](https://zenn.dev/tmasuyama1114/books/claude_code_basic/viewer/claude-md-hierarchy)

### グローバルルールの設定方法

設定手順はシンプルです。ターミナルで以下のコマンドを実行するだけでファイルを作成できます。

```
# フォルダを作成（すでにあれば不要）
mkdir -p ~/.claude

# ファイルを作成
touch ~/.claude/CLAUDE.md

# VS Codeで開いて編集
code ~/.claude/CLAUDE.md

# または vim で開いて編集
vim ~/.claude/CLAUDE.md
```

Windowsの場合は、PowerShellで以下のコマンドを実行してください。

```
# フォルダを作成
New-Item -Path "$HOME\.claude" -ItemType Directory -Force

# ファイルを作成
New-Item -Path "$HOME\.claude\CLAUDE.md" -ItemType File -Force
```

グローバルルールファイルに何を書けばいいか迷った場合は、別記事で詳しく解説しているので参考にしてみてください。

[【コピペOK】技術的負債を作らないためのルールを設定しよう（Claude Code, Codex, Cursor対応）](https://zenn.dev/tmasuyama1114/articles/global_rule_file)

## まとめ

この記事では、CLAUDE.mdの基本について解説しました。

- CLAUDE.mdはプロジェクトの文脈をClaude Codeに伝えるファイル
- `/init`コマンドで自動生成できる
- 具体的で簡潔な指示を記述することが効果的
- グローバルルール（`~/.claude/CLAUDE.md`）で全プロジェクト共通のルールも設定できる

まずは`/init`コマンドで生成してみてください。Claude Codeとの協働がより効率的になりますよ。

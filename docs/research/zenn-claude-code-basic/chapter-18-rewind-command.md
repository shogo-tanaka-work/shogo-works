# Claude Codeの/rewindコマンドで過去に戻る：会話の巻き戻しテクニック

Claude Codeで「さっきの方がよかった」「Claudeが変な方向に進んでしまった...」と思ったことはありませんか。

今回伝えたいことの要点は、`/rewind`**コマンドで会話を巻き戻せる**ということです。一言で言えば、「やり直したい」と思った瞬間に、過去の好きなポイントまで戻って別のアプローチを試せる機能です。

この記事では、`/rewind`の基本的な使い方から、`/clear`や`/compact`との違い、実践的な活用シーンまで解説します。読み終わる頃には、試行錯誤を効率化する方法が身についているはずです。

## 忙しい人のために要約

`/rewind`の要点を整理しました。「とりあえずこれだけ知りたい」という方はこちらをどうぞ。

- `/rewind`で会話を**過去の任意のポイントに巻き戻せる**
- ショートカットは`Esc`キー2回連続押し
- **3つの復元方法**があり、基本は「Restore code and conversation」を選ぶ
- bashコマンドや手動編集は追跡されない、30日で自動削除
- 巻き戻せるのは現在のセッション内で、`/clear`や`/compact`より前には戻れない
- 巻き戻した後に新しい指示を出すと、**別の分岐として進行**する

## /rewindコマンドとは

`/rewind`は、Claude Codeでの会話を**過去の特定のポイントまで巻き戻す**コマンドです。

「Claudeが良くない方向に進んでしまった」「最初の案の方がよかった」という場面、ありますよね。そんなとき、最初からやり直すのは面倒だと感じる方も多いのではないでしょうか。

`/rewind`なら、会話のチェックポイントを選択して、その時点まで状態を戻せます。つまり、気に入らない結果になっても、好きなポイントまで巻き戻して再出発できるわけです。

## チェックポイント機能とは

`/rewind`コマンドの背景にある「チェックポイント機能」について説明しておきましょう。

[https://docs.claude.com/ja/docs/claude-code/checkpointing](https://docs.claude.com/ja/docs/claude-code/checkpointing)

チェックポイント機能は、Claude Codeが行ったファイル編集を自動的に記録する機能です。いわば、タイムマシンのようなものだと考えるとわかりやすいかもしれません。

チェックポイント機能の仕組みや、AIが壊したコードの復旧について詳しく知りたい方は、別な記事で詳しく解説しています。

[Claude Codeの/rewindコマンド完全解説：AIが壊したコードを一発復旧](https://zenn.dev/tmasuyama1114/articles/claude_code_checkpoint_rewind)

すべてのプロンプト入力が自動的にチェックポイントとして保存されます。そのため、何か問題が起きたときに過去の状態に戻れるわけです。

### どんなときに便利?

実際の開発では、次のような場面で役立ちます。

まず、**異なる実装方法を試したいとき**。前の状態を保存しておけば、安心して実験できます。

また、**AIが予想外の変更をしてしまったとき**。すぐに元の状態に戻せるので、被害を最小限に抑えられます。

さらに、**複数のアプローチを比較したいとき**。それぞれの結果を確認してから、最適なものを選べます。

## 基本的な使い方

`/rewind`コマンドを実行すると、過去の会話ポイントの一覧が表示される仕組みです。

```
/rewind
```

### まだ何も変更していない場合

セッション開始直後など、まだ何も変更していない状態で`/rewind`を実行すると、次のように表示されます。

```
 Rewind

 Nothing to rewind to yet.

 Esc to exit
```

これは巻き戻せるポイントがまだないことを意味します。

### 編集後に実行した場合

Claude Codeにファイル編集を依頼した後で`/rewind`を実行すると、チェックポイントの一覧が表示されます。

```
 Rewind

 Restore the code and/or conversation to the point before…

   hello.txt に World と書いてください
   hello.txt +1 -1

 ❯ (current)

 Enter to continue · Esc to exit
```

矢印キーで戻りたいポイントに移動し、Enterを押すと確認画面に進みます。

### 複数のチェックポイントがある場合

複数回の編集を行った後は、それぞれのチェックポイントが一覧表示されます。

```
 Rewind

 Restore the code and/or conversation to the point before…

   hello.txt に hoge と追記してください
   hello.txt +1 -0

   fuga を追記して
   hello.txt +1 -0

   nyanko と追記して
   hello.txt +2 -1

 ❯ (current)
```

好きなポイントまで戻れるので、「3つ前の状態に戻したい」といった操作も簡単です。

### ショートカット：Escキー2回押し

「いちいちコマンドを打つのは面倒」と思った方もいるかもしれません。そんなときは、`Esc`キーを2回連続で押す方法がおすすめです。

この操作は、**直前のやりとりをキャンセルする**役割を果たします。Claudeの最新の応答が気に入らなかった場合、すぐに1つ前の状態に戻せるので便利です。

!

なお、`Esc`2回押しは直前のやりとりをすぐにキャンセルしたいときに最適です。一方で、複数のポイントから選んで戻りたい場合は、`/rewind`コマンドを使ってください。

## どこまで戻れるか

便利な`/rewind`ですが、巻き戻せる範囲には制限があります。「どこまででも戻れる」と思っていると困る場面があるかもしれないので、ここで確認しておきましょう。

具体的には、巻き戻せるのは以下の条件を満たす範囲内に限られます。

- **現在のセッション内**のポイントのみ
- `/clear`**を実行した地点**より後
- `/compact`**で圧縮されていない部分**

つまり、`/clear`でリセットした後や、`/compact`で要約された部分については、元の状態には戻れません。そのため、「ここは重要なチェックポイントだ」と思う場面があれば、`/clear`や`/compact`を実行する前に、必要なやりとりが完了しているか確認しておきましょう。

## 3つの復元方法

`/rewind`で巻き戻す際、以下の3つの復元方法から選択できます。

```
 Confirm you want to restore to the point before you sent this message:

 │ hello.txt に World と書いてください
 │ (2m ago)

 The conversation will be forked.
 The code will be restored +1 -1 in hello.txt.

 ❯ 1. Restore code and conversation
   2. Restore conversation
   3. Restore code

 ⚠ Rewinding does not affect files edited manually or via bash.
```

それぞれの動作と使いどころを見ていきましょう。

### 1. Restore code and conversation（推奨）

**コードと会話の両方を復元**します。デフォルトで選択されており、基本的にはこれを選ぶのがおすすめです。

この方法を選ぶと、ファイルの内容も会話履歴も、選択した時点の状態に戻ります。

- **Claude Codeの認識**：選択した時点までの内容のみ
- **実際のファイル**：選択した時点の状態

両方が一致するため、最もわかりやすい復元方法です。プロンプトを修正して再実行したい場合に最適です。

### 2. Restore conversation

**会話のみを復元**し、コードの変更は維持されます。

たとえば、Claude Codeが複数ファイルを編集して、一部の変更だけ残したい場合に使えます。会話を巻き戻してから、手動で不要な変更を元に戻すことで、必要な変更だけを残せます。

ただし、この方法には注意が必要です。

- **Claude Codeの認識**：選択した時点までの内容のみ
- **実際のファイル**：変更されたまま

Claude Codeの認識とファイルの状態にズレが生じるため、次の指示を出すときに予期しない動作をする可能性があります。

### 3. Restore code

**コードのみを復元**し、会話履歴は維持されます。

Claude Codeの変更は気に入らなかったが、会話履歴を見返したい場合に使えます。どこで判断を誤ったのかを確認しながら、別のアプローチを試せます。

ただし、こちらもClaude Codeの認識とファイルの状態にズレが生じます。次の指示を出す際は、「先ほどの変更を巻き戻した」ことを明示的に伝える必要があります。

### 結論：基本は「Restore code and conversation」

特別な理由がない限り、**Restore code and conversation** を選びましょう。Claude Codeの認識とファイルの状態が一致するため、予期しない動作を防げます。

## 活用シーン

`/rewind`が効果を発揮する場面を紹介します。実際の開発で遭遇しやすいケースばかりなので、参考にしてください。

### Claudeの提案が期待と違ったとき

Claudeが生成したコードやアイデアが期待と違う、というのはよくあることです。そんなときは`/rewind`で戻ってから、別の指示を出してみましょう。

例えば、リファクタリングの方向性が違った場合の手順はこうなります。

```
1. Claudeにリファクタリングを依頼
2. 結果が思っていたものと違う
3. Esc 2回（または /rewind）で戻る
4. 「関数を分割するのではなく、命名を改善してほしい」と再指示
```

試行錯誤を効率化するために、ぜひ覚えておいてください。

### 別のアプローチを試したいとき

1つのアプローチを試した後、「やっぱり別の方法でやってみたい」と思うことってありますよね。

そんなときも`/rewind`でアプローチを選択する前の状態に戻り、異なる方針で進められます。複数のアプローチを比較検討できるのがメリットです。

### 誤った指示を出してしまったとき

「しまった、意図と違う指示を出してしまった」ということ、ありませんか。

Claudeがそのまま作業を進めてしまうと、無駄な時間が発生してしまいます。

そんなときも`/rewind`で誤った指示を出す前の状態まで戻り、正しい指示を出し直せます。指示ミスをリカバリーできるのは心強いですよね。

### 複数の実装を比較したいとき

ちょっと高度な使い方ですが、複数のアプローチを比較したいときにも活用できます。

まず1つ目の実装を試して、チェックポイントで戻ります。次に2つ目の実装を試して、結果を比較します。最終的に、より良い方を選択できます。

ただし、`Restore code and conversation`を選ぶと、1つ目の実装もリセットされてしまいます。

残しておきたい場合には、GitのStashを使って変更を退避させておくと良いでしょう。

```
# 1つ目の実装を退避
git stash

# /rewindで戻って2つ目の実装を試す
# 比較後、必要に応じてstashを適用
git stash pop
```

複数の実装を並列で試したい場合は、Git worktreeを活用する方法もあります。

[世界一わかりやすくGit worktreeを解説！AI駆動開発でも活用できる並列開発の方法](https://zenn.dev/tmasuyama1114/articles/git_worktree_beginner)

### 実験的な変更を試すとき

新しいアプローチを試す前に、チェックポイントがあることを確認します。

たとえば「この設計で動くかな？」というような実験的な変更を試すときに便利です。もし失敗しても、すぐに元に戻せますので、大胆な変更に挑戦できますね。

会話も巻き戻せるので、コンテキストが汚れる心配もありません。最初はとりあえずざっくり指示を出して、うまくいかなければ巻き戻して修正するといった活用ができます。

## 制限事項

`/rewind`を使う際に知っておくべき制限があります。事前に理解しておけば、「思っていたのと違う」という事態を避けられますよ。

### bashコマンドの変更は追跡されない

Claude Codeがbashコマンドでファイルを変更した場合、チェックポイントには記録されません。

たとえば、Claude Code の外で、自分でファイルを編集した場合、チェックポイントには記録されません。

```
# 自分でファイルを編集した例
rm file.txt
mv old.txt new.txt
cp source.txt dest.txt
```

これらの変更は、巻き戻しの対象外です。あくまで、Claude Code のファイル編集ツール（Read/Write/Edit）を使った変更のみが追跡されます。

### 手動編集は追跡されない

上記と同じく、エディタで直接ファイルを編集した場合も、チェックポイントには記録されません。

Claude Code外での変更は、チェックポイントの対象外です。ただし、Claude Codeが同じファイルを編集すれば、その時点の状態は記録されます。

たとえばCursorとClaude Codeを並行して使う場合、Cursorでファイルを編集してもチェックポイントには記録されません。巻き戻し機能を活用するなら、Claude Codeに指示を出してファイルを編集しましょう。

### チェックポイントは30日後に自動削除

チェックポイントは永続的なものではありません。30日後に自動的に削除されます。

長期的なバージョン管理には、Gitなどを使用すべきです。チェックポイント機能は、あくまでセッション内での素早い復元が目的であり、Gitを補完する機能として考えましょう。

### 分岐の概念

「巻き戻した後、元の流れに戻れるの？」と疑問に思う方もいるかもしれません。

結論から言うと、**元の未来には戻れません**。

一度巻き戻した後に新しい指示を出すと、そこから別の分岐として会話が進行します。

```
A → B → C → D（元の流れ）
        ↓
        C'（/rewindで戻ってから別の指示）→ D'（新しい流れ）
```

Gitのブランチに似た仕組みです。過去の状態から新しい分岐を作るイメージで捉えるとわかりやすいのではないでしょうか。

### /clearや/compactとの併用

`/clear`や`/compact`を実行すると、それより前のポイントには戻れなくなります。これは重要なポイントなので、しっかり覚えておいてください。

それぞれの違いを整理しておきましょう。

- **`/clear`**：会話履歴を完全にリセットするため、それより前には戻れない
- **`/compact`**：会話を要約・圧縮するため、圧縮された部分の詳細には戻れない

したがって、「ここまでの作業は確定」という場合は`/clear`や`/compact`を使い、「まだ試行錯誤中」という場合は`/rewind`で戻れる状態を維持する、という使い分けが効果的です。

## /rewindと他のコマンドの比較

「`/rewind`と`/clear`、`/compact`って何が違うの？」と思う方もいるかもしれません。ここで整理しておきましょう。

| コマンド | 動作 | 用途 |
| --- | --- | --- |
| `/rewind` | 過去のポイントに巻き戻す | 別のアプローチを試したい |
| `/clear` | 会話履歴を完全リセット | タスクを完全に切り替えたい |
| `/compact` | 会話を要約して圧縮 | 文脈を維持しつつトークン節約 |

まとめると、`/rewind`は「**試行錯誤**」のためのコマンドとして機能します。

一方、`/clear`は「リセット」の役割を担い、`/compact`は「整理」のために使うという違いがあります。

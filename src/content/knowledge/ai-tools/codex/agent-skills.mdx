## TL;DR

Codex CLI の Agent Skills は、**繰り返し使うワークフローを「スキル」として定義・再利用できる機能**です。2025 年に追加された機能で、CLI とすべての IDE 拡張（VS Code・Cursor・Windsurf）で共用できます。

Claude Code の Skills と似た概念ですが、スクリプトファイルや設定ファイルを一緒にバンドルできる点が特徴です。

---

## Agent Skills の保存場所

```
~/.codex/skills/<skill-name>/           ← 個人グローバル定義
  skill.md                              ← スキル定義（必須）
  setup.sh                              ← セットアップスクリプト（任意）
  resources/                            ← 追加リソースファイル（任意）

.codex/skills/<skill-name>/             ← プロジェクト定義（チーム共有）
  skill.md
  scripts/
  templates/
```

## skill.md の基本構造

```markdown
---
name: run-tests
description: プロジェクトのテストスイートを実行して結果をレポートする
approval_required: false
sandbox_mode: workspace-write
---

プロジェクトのテストを実行してください。

1. まず `package.json` を確認してテストコマンドを特定する
2. テストを実行する（`npm test` / `pnpm test` / `yarn test` のいずれか）
3. 失敗しているテストがあれば、原因と修正方針をレポートする
4. カバレッジレポートがあれば、低いカバレッジの箇所を報告する
```

## frontmatter フィールド一覧

| フィールド | デフォルト | 説明 |
|---|---|---|
| `name` | ファイル名 | スキルの識別名（呼び出し名） |
| `description` | — | スキルの目的と使うタイミング（Codex が自動判断に使う） |
| `approval_required` | `true` | `false` にすると確認なしに自動実行 |
| `sandbox_mode` | セッションのデフォルト | このスキル専用のサンドボックスモード |
| `tools` | 全ツール | 使用を許可するツールのリスト |
| `model` | セッションのデフォルト | このスキル用のモデル |
| `model_reasoning_effort` | `"medium"` | 推論レベル |
| `context` | `"inline"` | `"fork"` で独立コンテキスト実行 |

## approval_required の使い分け

```markdown
---
name: format-code
description: コードのフォーマットを整える（副作用が少ない操作）
approval_required: false    # 自動実行OK
---

`prettier` でコードをフォーマットしてください。
```

```markdown
---
name: deploy-production
description: 本番環境にデプロイする（不可逆な操作）
approval_required: true     # 必ず確認を求める
---

本番環境へのデプロイを実行します。実行前に変更内容を確認してください。
```

## context: fork — 独立コンテキストで実行

`context: fork` を指定すると、スキルがメインセッションとは独立したコンテキストで実行されます。長時間かかるタスクや、メインのコンテキストを汚したくない場合に使います。

```markdown
---
name: security-audit
description: コードベースのセキュリティ監査（独立コンテキストで実行）
context: fork
sandbox_mode: read-only
tools: ["Read", "Glob", "Grep"]
model_reasoning_effort: high
---

コードベース全体をセキュリティ観点で監査してください。

対象：
- 認証・認可ロジック
- SQL クエリ（インジェクションリスク）
- 入力バリデーション
- 機密情報のハードコード
- 外部 API の安全な呼び出し

変更は行わない。発見した問題を Markdown レポートとして出力する。
```

## スクリプトのバンドル

スキルディレクトリにスクリプトを置いて一緒にバンドルできます。

```bash
# ディレクトリ構造
.codex/skills/check-deps/
├── skill.md         ← スキル定義
├── check-npm.sh     ← npm チェックスクリプト
├── check-yarn.sh    ← yarn チェックスクリプト
└── report.py        ← レポート生成スクリプト
```

```markdown
---
name: check-deps
description: 依存関係の脆弱性・ライセンスチェックを実行する
sandbox_mode: workspace-write
---

依存関係のチェックを実行してください。

1. `.codex/skills/check-deps/check-npm.sh` を実行して npm audit の結果を取得する
2. `.codex/skills/check-deps/report.py` で結果をフォーマットする
3. CRITICAL・HIGH の脆弱性があれば修正方法を提示する
```

## テンプレートのバンドル

よく使うファイルテンプレートをスキルに含めることができます。

```bash
.codex/skills/create-feature/
├── skill.md
└── templates/
    ├── service.ts.template
    ├── controller.ts.template
    ├── dto.ts.template
    └── service.spec.ts.template
```

```markdown
---
name: create-feature
description: NestJS の機能モジュールをテンプレートから生成する
---

`$ARGUMENTS` という名前の NestJS 機能モジュールを作成してください。

1. `.codex/skills/create-feature/templates/` のテンプレートを使う
2. テンプレート内の `FEATURE_NAME` を `$ARGUMENTS` で置換する
3. 生成したファイルをテストして動作することを確認する
```

## 実践的なスキル例

### コミットメッセージ生成

```markdown
---
name: smart-commit
description: ステージングされた変更から Conventional Commits 形式のコミットメッセージを生成してコミットする
approval_required: true
---

ステージングされている変更を分析して、適切なコミットメッセージを生成しコミットしてください。

1. `git diff --staged` で変更内容を確認する
2. 変更の種類に応じて prefix を選ぶ: feat / fix / refactor / test / docs / chore
3. コミットメッセージは `prefix(scope): 件名` の形式（日本語）
4. コミット前に生成したメッセージを提示して確認を求める
```

### PR レビューサマリー生成

```markdown
---
name: pr-summary
description: 現在のブランチの変更内容からPRの説明文を生成する
approval_required: false
context: fork
---

現在のブランチ（`git diff main...HEAD`）の変更内容を分析して、
PR の説明文を生成してください。

生成する項目：
- タイトル（1行・70文字以内）
- 変更の概要（3〜5点の箇条書き）
- テスト方法
- スクリーンショットが必要か（UI 変更がある場合）

GitHub の PR 本文として使える Markdown 形式で出力してください。
```

### 型エラー自動修正

```markdown
---
name: fix-types
description: TypeScriptの型エラーを自動修正する
approval_required: false
tools: ["Read", "Edit", "Bash(npx tsc *)"]
---

TypeScript の型エラーを修正してください。

1. `npx tsc --noEmit 2>&1` で型エラーの一覧を取得する
2. エラーを解析して修正方針を立てる
3. `any` や `as unknown` でキャストして誤魔化さない（型を正しく定義する）
4. 修正後に再度 `npx tsc --noEmit` を実行して全エラーが解消されたか確認する
```

## IDE 拡張での使い方

VS Code・Cursor・Windsurf の Codex 拡張では、コマンドパレットからスキルを呼び出せます。

```
Ctrl+Shift+P / Cmd+Shift+P
→ "Codex: Run Skill" を選択
→ スキル名をタイプして選択
```

CLI とプロジェクトの `.codex/skills/` は共有されているため、ターミナルと IDE で同じスキルが使えます。

## Claude Code Skills との比較

| 観点 | Codex Agent Skills | Claude Code Skills |
|---|---|---|
| フォーマット | Markdown + TOML frontmatter | Markdown + YAML frontmatter |
| スクリプトバンドル | ディレクトリにスクリプトを同梱可能 | スキルファイル 1 枚のみ |
| テンプレートバンドル | ディレクトリにテンプレートを同梱可能 | 非対応 |
| IDE 統合 | VS Code / Cursor / Windsurf で共有 | Claude Code の全サーフェスで共有 |
| 引数 | `$ARGUMENTS`（同様） | `$ARGUMENTS`、`$1`、`$2`... |
| 独立コンテキスト | `context: fork` | `context: fork` |

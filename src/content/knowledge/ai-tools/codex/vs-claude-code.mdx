## TL;DR

Codex CLI と Claude Code は「ターミナルで動く AI コーディングエージェント」という点では同じカテゴリですが、**設計哲学とアーキテクチャが根本的に異なります**。

- **Codex CLI**: 「自律的に動かして後でレビュー」。OS レベルのサンドボックスで安全を確保し、並列タスク・CI/CD 連携が得意。OSS（Apache 2.0）。
- **Claude Code**: 「逐次確認しながら一緒に作る」。コンテキスト理解と精度が高く、1M トークンの大規模コードベース処理が得意。クローズドソース。

どちらかが「優れている」のではなく、**タスクの性質とチームのワークスタイルによって使い分け**が最適です。

## 設計哲学の違い

### Codex CLI：「自律実行モデル」

```
タスク指示 → エージェントが自律的に実行 → 結果を後でレビュー
```

- 人の介在を最小化することでスループットを最大化
- OS サンドボックスが安全ネットになるため、承認なしで動かしやすい
- CI/CD での完全自動化を想定した設計

### Claude Code：「協調作業モデル」

```
タスク指示 → Claude が確認・計画を提示 → 人が承認 → 実行
```

- 人と Claude が対話しながら進めることを前提
- Plan Mode で事前確認を促す設計
- 複雑な意思決定を人が担うことを前提としたフロー

## 安全モデルの比較

| 観点 | Codex CLI | Claude Code |
|---|---|---|
| サンドボックス | OS レベル（sandbox-exec / bwrap） | なし（Hooks でのソフト制限） |
| ネットワーク制御 | OS 制限（デフォルト無効） | 制限なし（Hooks でブロック可） |
| ファイルアクセス制限 | OS パス制限で強制 | `allowedDirectories` 設定で制限 |
| 危険コマンドブロック | OS サンドボックスで強制 | PreToolUse Hook で実装 |
| 承認モード | 4 種類（on-request/never/untrusted/granular） | 4 種類（normal/acceptEdits/plan/autoAccept） |

**Codex CLI が有利な場面：** OS レベルの強制力が必要な場面（信頼できないコードの実行、共有 CI 環境）

**Claude Code が有利な場面：** 柔軟なカスタマイズが必要な場面（プロジェクト固有のブロックルール、複雑な権限管理）

## エージェント機能の比較

### サブエージェント

| 観点 | Codex CLI | Claude Code |
|---|---|---|
| 定義ファイル形式 | TOML | Markdown（YAML frontmatter） |
| 推論レベル設定 | `model_reasoning_effort: low/medium/high` | なし（モデル変更で代替） |
| サンドボックス指定 | エージェント単位で `sandbox_mode` を設定 | `isolation: worktree` のみ |
| スレッド切り替え | `/agent` コマンドでセッション中に切り替え | 切り替えなし（メインのみ） |

### マルチエージェント

| 観点 | Codex CLI | Claude Code |
|---|---|---|
| エージェント間通信 | メールボックス経由 | Agent Teams（実験的機能） |
| 共有タスクリスト | あり | Agent Teams のみ |
| 並列スレッド数制限 | `max_threads` で設定 | 制限設定なし |
| 安定性 | 安定版 | Agent Teams は実験的 |

## プロジェクト設定の比較

| 観点 | Codex CLI | Claude Code |
|---|---|---|
| 設定ファイル名 | `AGENTS.md` | `CLAUDE.md` |
| 設定形式 | プレーン Markdown | Markdown + frontmatter（path-scoped rules） |
| インポート | なし | `@path/to/file` でインポート可能 |
| 上書き機能 | `AGENTS.override.md` | 下位スコープが上書き |
| 自動学習 | なし | Auto Memory（MEMORY.md）で自動蓄積 |
| 代替ファイル名 | `config.toml` で変更可 | 変更不可（`CLAUDE.md` 固定） |

**Claude Code の CLAUDE.md が有利な点：** `@インポート` による分割管理、path-scoped rules によるファイル種別ごとの設定、Auto Memory による自動学習

**Codex CLI の AGENTS.md が有利な点：** `AGENTS.override.md` による完全置換、サブディレクトリ自動マージ、チームの既存ドキュメントファイルへのリネーム対応

## CI/CD 連携の比較

| 観点 | Codex CLI | Claude Code |
|---|---|---|
| 公式 GitHub Action | `openai/codex-action@v1`（公式） | 公式提供なし |
| CI でのセットアップ難度 | Action 1 行で完結 | Agent SDK スクリプトの実装が必要 |
| OS サンドボックスの有無 | あり（安全な CI 実行） | なし（Hooks でのソフト制限のみ） |
| 承認不要の自動実行 | `approval-policy: never` で完全自動 | `autoAccept` モードで完全自動 |

**Codex CLI が有利な場面：** CI/CD での完全自動化、OS レベルのセキュリティが必要な環境

**Claude Code が有利な場面：** 既存の Python/TypeScript スクリプトとの統合（Agent SDK）、細かいカスタマイズが必要な場合

## ユースケース別推奨

### Codex CLI が向いているケース

- **完全自動化パイプライン**: CI/CD での承認なし実行、Issue→PR の自動生成
- **信頼できないコードの安全な実行**: OS サンドボックスが必須の環境
- **コスト最適化重視**: GPT-5.4-mini + 低推論レベルでトークンコストを最小化
- **既存 OpenAI エコシステム**: Agents SDK・Realtime API を使っているチーム
- **GitHub Action との統合**: 公式アクションで簡単にセットアップしたい

### Claude Code が向いているケース

- **大規模コードベースの理解**: 1M トークンで 25,000 行以上を一括処理
- **精度重視の実装**: SWE-bench スコアが重要なタスク
- **対話型の協調作業**: Plan Mode で計画を確認しながら進めるスタイル
- **CLAUDE.md の高度な管理**: path-scoped rules・@インポート・Auto Memory を活用
- **チームの知識蓄積**: Auto Memory でプロジェクト固有の知識を継続的に蓄積

### 両方使うハイブリッド構成

実務では使い分けが最も効果的です：

```
【日常的な開発作業】
Claude Code（対話型・精度重視）

【CI/CD での自動化】
Codex CLI（完全自動・GitHub Action）

【大規模コードベースの調査】
Claude Code（1M コンテキスト）

【並列タスクの高速処理】
Codex CLI（効率的なトークン使用・並列スレッド）
```

## 2026年の進化トレンド

### Codex CLI の方向性
- Rust 実装のパフォーマンス優位性を活かした高速処理
- GitHub Action・Agents SDK との統合強化
- MCP Apps のリッチな機能拡張（ファイルアップロード・サーバー駆動エリシテーション）
- Windows サポートの成熟（v0.118.0 以降でネットワーク制御を強化）

- Agent Teams の安定版リリースに向けた開発継続
- CLAUDE.md・Auto Memory によるプロジェクト知識の自動蓄積
- Agent SDK による他プラットフォーム（Bedrock・Vertex AI）連携の強化
- VS Code 拡張のネイティブ統合（Web 版との機能統一）

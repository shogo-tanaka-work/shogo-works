## 教材化ポイント

### ツール検索のデフォルト化 — 「選択肢が多すぎる」問題への回答

MCP ツールが tool search ベースに変わったことは、「ツールが増えすぎると全リストから選ぶのは難しい」という問題への対応。人間の認知負荷と同様に、LLM も選択肢が多すぎると精度が落ちる（lost-in-the-middle 問題）。

「選択肢を検索可能にする」というアプローチは、検索エンジン・コマンドパレット・RAG など、人間が使う UI 設計でも同じ発想。AI のツール選択と人間の情報探索は同じ課題を持つという視点で教材化できる。

### macOS プロキシ対応 — 「当然できるはずの機能」が後から追加される理由

企業環境では PAC/WPAD が標準的なプロキシ設定方法だが、Codex がそれを「サポートしていなかった」ことはツール開発の典型的な優先度問題を示す。

「コア機能が動くようになったら、次に企業環境の前提条件（プロキシ・SSO・証明書管理）を整備していく」というプロダクト成熟の順序は、B2B SaaS の開発ロードマップの説明として使える。

### safety-buffering UI — 「危険な操作の前に確認を挟む」設計原則

危険な操作をバッファリングして UI で確認を求める仕組みは、「人間を最終判断者として残す」設計思想の実装。AI が自動実行できる範囲と人間の確認が必要な範囲を分ける設計は、エージェント AI のリスク管理において中心的なテーマ。

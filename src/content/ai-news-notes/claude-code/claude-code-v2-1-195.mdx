## 教材化ポイント

### hook matcher の部分一致バグ — 「意図した設計」と「実装の乖離」

`code-reviewer` という hook matcher が `reviewer` という文字列を含む全識別子にマッチしてしまったバグは、「文字列マッチングの実装ミスが予期しない副作用を生む」典型例。

正規表現や glob パターンの部分一致 vs 完全一致の違いはセキュリティにも影響する（例：`admin` という matcher が `not-admin` にもマッチしてしまう問題）。ホワイトリスト設計では「完全一致がデフォルトであるべき」という原則の実例として使える。

修正後のワイルドカード記法（`mcp__brave-search__.*`）は「明示的にワイルドカードを書かないと広くマッチしない」という設計哲学を示す。「デフォルト安全（secure by default）」の考え方。

### 日本語音声入力 auto-submit — ロケール依存バグの見つけ方

「スペースなし言語では auto-submit が動かない」というバグは、英語基準で開発・テストしていると見つかりにくい典型的なロケール依存バグ。

日本語・中国語・タイ語はスペースなしで文が続く → 「単語の終わり（=スペース）を区切りにした送信判定」が機能しない。多言語対応開発における「自分の言語でしかテストしない」落とし穴を教材として使える。

Claude Code チームが修正したことは、「日本語ユーザーのフィードバックが開発チームに届いている」証拠でもある。OSS リリースノートを読む理由として「バグ修正から製品の課題を知る」視点を紹介できる。

### バックグラウンドエージェントの「到達不能デーモン」バグ

control socket の起動失敗でデーモンが到達不能のまま動き続けてリスタートをブロックするバグは、分散システムの「ゾンビプロセス」問題の AI エージェント版。

`kill -9` 相当の強制終了なしに復旧できない状態になる問題は、エージェント基盤のロバスト性設計の難しさを示す。「ヘルスチェック付きプロセス管理」「グレースフルシャットダウン」という設計概念と結びつけて教材化できる。

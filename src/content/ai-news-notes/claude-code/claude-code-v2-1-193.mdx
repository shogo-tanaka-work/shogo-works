## 教材化ポイント

### OTel 応答ログのデフォルトリダクション設計

`claude_code.assistant_response` が追加されたが、デフォルトはリダクション（非記録）になっている。「ログを取れる技術があっても、デフォルトはプライバシー保護側にする」という設計判断は、エンタープライズ向けツールの典型的な原則だ。

`OTEL_LOG_USER_PROMPTS` が有効な環境では自動的に応答もログされる仕様（暗黙の継承）は注意点として教材化できる。「設定が想定外の動作を引き起こすケース」として、アップデート時の環境変数確認の重要性を伝えるのに使える。

### autoMode.classifyAllShell — 安全性と利便性のトレードオフ

「全コマンドをチェックするか、怪しいコマンドだけチェックするか」という設計は、セキュリティとパフォーマンスのトレードオフの典型例。デフォルトを絞りつつ opt-in で拡大する構成は、Claude Code の auto-mode 全体の設計哲学と一致している。

「なぜデフォルトで全部チェックしないのか」という問いから、AI エージェントのレイテンシ・スループットへの影響を議論できる。

### MCP headersHelper の自動再接続 — 分散システムの retry パターン

ツール呼び出しが 401/403 を返したら headersHelper を再実行して再接続するという仕組みは、「一時的な認証エラーを一次エラーとして扱うのでなく、再認証して回復する」設計パターン。

OAuth の token refresh と同じ発想で、長時間実行するエージェントでトークン有効期限切れが起きても止まらない設計。CI/CD の retry ロジックや API クライアントの exponential backoff と並べて教材化できる。

### ファントムエージェントのバグ — 状態管理の難しさ

メインターンをバックグラウンド化したときにファントムの「general-purpose (resumed)」が生成され、メイン会話を再実行するバグは、マルチエージェントの状態管理の複雑さを示す実例。「エージェントが自分自身のコピーを作ってしまう」という問題は、分散エージェントシステムの設計課題として興味深いトピック。

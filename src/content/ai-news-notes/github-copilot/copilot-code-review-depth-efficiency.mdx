## 教材化ポイント

### カスタムツールを廃止してビルトインを使う — 「標準ツールの方が速い」設計判断

カスタムファイルツールを捨てて `grep`・`rg`・`glob`・`view` に移行することで20%のコスト削減を達成した。これは「特注ツールより枯れた標準ツールの方が効率的だった」という発見であり、AI ツール開発における試行錯誤の産物。

「最初は専用ツールを作りたがるが、後から標準ツールで十分と気づく」というパターンはソフトウェア開発全般に通じる。不必要な抽象化を避ける原則（YAGNI/KISS）の実例として使える。

### 分析深度（Depth）の段階設計 — 品質と速度のスライダー

Low（GA）と Medium（Preview）という2段階のデプスは、「精度と速度・コストのトレードオフを利用者が選べる」設計。ユーザーが状況に応じて最適なバランスを選べる UI 設計は AI ツール全般に広がりつつある。

「モデルサイズの選択（Haiku / Sonnet / Opus）」と同じ構造で説明できる。より深い分析はより高コスト・時間がかかるが、より多くの問題を見つける。

### 組織デフォルト設定 — 「標準を決めることの重要性」

組織全体にデフォルトを設定し、個別リポジトリがオーバーライドできる設計は、設定管理の階層化（階層的なポリシー）の実装。GitLab の Inheritance Policy や AWS Organizations の SCP と同じ発想。

「チーム全員が同じツールを同じ設定で使う」環境を作ることのコスト効果（ナレッジ共有・ベストプラクティスの定着）として教材化できる。

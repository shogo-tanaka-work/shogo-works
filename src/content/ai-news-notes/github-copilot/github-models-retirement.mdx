## 教材化ポイント

### 1. 実験的サービスの「本番統合」による廃止パターン

GitHub Models は「試せる場所」として始まったが、GitHub Copilot と Azure AI Foundry という本番サービスが成熟した段階で廃止される。これは「PoC → 本番移行」後に実験環境が役割を終えるパターン。

**教材への落とし込み方:**
- 企業の AI 導入プロセス（PoC環境 → 本番サービスへの昇格）の文脈で使える
- 「実験基盤と本番基盤は別に設計しておく」という教訓の実例

### 2. 移行先の選択フレームワーク

Azure AI Foundry（本格API・管理機能）vs GitHub Copilot（IDE統合・開発者体験）という選択肢は、「何のためにAIを使うか」によって適切な移行先が変わるという判断例。

**教材への落とし込み方:**
- AI ツール選定の「ユースケース起点のフレームワーク」として使える
- プロトタイプ用途 vs 本番API用途 vs 開発補助用途の分類

### 3. 廃止告知から実施まで4週間という短期間

2026-07-01 告知 → 2026-07-30 廃止は約4週間。この短さは「依存度が高いサービスほど早めの移行計画が重要」という教訓として使える。

**教材への落とし込み方:**
- SaaSサービスへの依存リスク管理の例として使える
- 「廃止リスクを折り込んだ技術選定」というポイントの説明材料

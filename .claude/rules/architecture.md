# アーキテクチャルール

## 設計原則
- 1モジュール1責務。複数の責務を持つファイルは分割する
- 依存の方向は一方向に保つ（UI → ロジック → データ）
- 外部サービスへのアクセスは必ずラッパーを介する
  （直接 SDK を呼ばない。テスト・差し替えを容易にするため）

## ファイル構成の基本方針
- 機能ごとにディレクトリを切る（feature-based structure）
- 共通ロジックは lib/ または utils/ に集約する
- 型定義は types/ または各機能ディレクトリの types.ts に集約する

## エラーハンドリング
- すべての非同期処理は try/catch で囲む
- エラーはユーザーに適切なメッセージで返す（内部エラーの露出禁止）
- ログは必ず残す（console.log ではなく logger 経由）

## 環境変数
- 環境変数は必ず検証してから使う（起動時に未設定を検出する）
- .env はコミットしない。.env.example を必ず用意する
